# Тикет 2: Настройка базовой архитектуры маршрутизации (Vue Router) и управления состоянием (Pinia)

## Описание

Необходимо заложить фундамент клиентской логики: настроить роутинг для перемещения между экранами и подключить глобальный стейт-менеджер для хранения данных сессии (информация из внешней CRM, роли).

## Технические требования

- **Стек:** Vue Router 4, Pinia.
- **Маршрутизация:** Настроить два основных слоя (layouts):
    - `AuthLayout` (для страницы логина/заглушки авторизации).
    - `AppLayout` (основной интерфейс с навигацией для авторизованных пользователей).
- **Управление состоянием:** Создать `useAuthStore` (Pinia), который будет хранить:
    - `token` (строка; в mock-режиме формат `mock:<ключ-роли>:<employee_ext_id>`, см. Тикет 37).
    - `userRole` (одна из констант `USER_ROLES` в `stores/auth.ts`).
    - `userInfo` (`id`, `fullName`, `email` — для шапки и аудита на бэкенде).
- **Route Guards:** В `router.beforeEach`: восстановление сессии из `localStorage` (`auth.token`, `auth.role`, `auth.userInfo`); при `meta.requiresAuth` без валидной пары токен+роль — редирект на именованный роут `login`; при заходе на `/` или `login` с валидной сессией — редирект на дефолтный маршрут роли через `getDefaultRouteForRole` из `src/utils/roleRoutes.ts` (менеджер → `orders`, мастер → `workshop-tasks`, и т.д.); при несовпадении роли с `meta.allowedRoles` — редирект на дефолт роли; catch-all `/:pathMatch(.*)*` → `NotFoundView` без авторизации.

## Критерии приемки (DoD)

1. Пакеты `vue-router` и `pinia` установлены и подключены в `main.ts`.
2. В папке `src/router/index.ts` создана базовая конфигурация маршрутов.
3. В папке `src/stores/auth.ts` реализован стор для авторизации.
4. Навигационные хуки (guards) корректно защищают внутренние маршруты; предусмотрен catch-all на `NotFoundView`.

# Тикет 3: Установка и базовая настройка библиотеки UI-компонентов

## Описание

Для обеспечения высокой скорости разработки и чистоты кода необходимо подключить и настроить UI-библиотеку / CSS-фреймворк. Согласно требованиям легковесности, оптимальным выбором будет **Tailwind CSS** (или легковесная компонентная база поверх него, например Headless UI или Element Plus, по договоренности с командой).

## Технические требования

- **Стек:** Tailwind CSS (PostCSS, Autoprefixer) + плагины (forms, typography) ИЛИ Element Plus (на выбор архитектора). _Рекомендуется Tailwind CSS для гибкости._
- **Иконки:** Подключить библиотеку иконок (например, `@iconify/vue` или `heroicons`), которые понадобятся для сайдбара и кнопок.
- **Глобальные стили:** Настроить `tailwind.config.js` под бренд (определить базовые цвета: primary, secondary, цвета статусов заказа — success, warning, danger).

## Критерии приемки (DoD)

1. Установлены необходимые зависимости UI-фреймворка.
2. Конфигурационные файлы (например, `tailwind.config.js`, `postcss.config.js`) настроены корректно.
3. В `src/assets/style.css` (или аналоге) импортированы базовые директивы.
4. Создан тестовый компонент `HelloUI.vue` для проверки корректности применения стилей (кнопка с primary цветом).

# Тикет 4: Создание базового layout-компонента (навигация, сайдбар, шапка)

## Описание

Создание основного каркаса приложения (App Layout), внутри которого будут отображаться все рабочие экраны системы. Layout должен адаптироваться под роль пользователя (показывать только нужные пункты меню).

## Технические требования

- **Структура компонента (`src/layouts/AppLayout.vue`):**
    - **Sidebar (боковая панель):** Список `menuItems` с именованными `router-link` по роли: для **Менеджер** — «Мои заказы» (`orders`), «Заявки с сайта» (`manager-applications`); **Мастер цеха** — сменные задания; **Кладовщик** — остатки и транзакции; **Закупщик** — дефицит; **Монтажник** — выезды; **Руководитель** — мониторинг, загрузка линий, рентабельность. На мобильных — выезжающий сайдбар с оверлеем.
    - **Доп. блок для менеджера на `order-detail`:** В сайдбаре при открытой карточке заказа — секция «Управление заказом»: быстрая смена стадии (кнопки, согласованные с правилами `StatusManager` на бэкенде), кнопка редактирования (модалка `OrderFormModal`), отмена заказа (`DELETE`) с подтверждением; вызовы через `useManagerOrdersStore` (`updateOrderStatus`, `updateOrder`, `deleteOrder`).
    - **Header (шапка):** ФИО, роль, `userInfo.id` (employee_ext_id), кнопка «Выйти» (`router.push` на `/login`).
    - **Main Content Area:** `<router-view />`.
    - **Переиспользуемые таблицы:** при необходимости плотных списков — компонент `src/components/Common/AppDataTable.vue` (колонки, сортировка, пагинация, `fitContainer` / `density`).
- **Адаптивность:** Базовая отзывчивость — сайдбар должен скрываться или сворачиваться на мобильных экранах (важно для Монтажников с планшетами/телефонами).

## Критерии приемки (DoD)

1. Компонент `AppLayout.vue` создан и сверстан.
2. Шапка отображает данные из Pinia (`useAuthStore`).
3. При нажатии на кнопку «Выйти» токен очищается, происходит редирект на экран авторизации.
4. Боковое меню фильтруется на основе роли пользователя (например, вкладку «Загрузка цеха» видит только Руководитель).

# Тикет 5: Реализация экрана-заглушки авторизации (подготовка к интеграции с CRM)

## Описание

Поскольку реальная аутентификация в системе будет делегирована внешней корпоративной CRM (по спецификации С4), нам нужен экран-заглушка для локальной разработки и тестирования, который позволит легко переключаться между ролями сотрудников.

## Технические требования

- **Компонент (`src/views/Auth/LoginView.vue`):**
    - Два селекта: **роль** (`USER_ROLES`) и **сотрудник** (`MOCK_EMPLOYEES`, отфильтрованный по выбранной роли) — значение второго селекта есть `employee_ext_id`, уходит на бэкенд как идентификатор текущего пользователя.
    - Кнопка «Войти» вызывает `authStore.login(selectedRole, selectedEmployeeId)`.
    - Опционально: сохранение последнего выбора в `localStorage` под ключом `myworkshop:last-login` и восстановление при следующем визите.
- **Логика Store:** `login(role, employeeExtId)` формирует токен `mock:<ROLE_TOKEN_KEYS[role]>:<employeeExtId>` (например `mock:manager:crm-manager-001`), сохраняет `auth.token`, `auth.role`, `auth.userInfo` (JSON) в `localStorage` и обновляет Pinia.
- **Навигация:** После входа — `router.push(getDefaultRouteForRole(role))` на именованный маршрут роли, а не на абстрактный `/`.

## Критерии приемки (DoD)

1. Страница `LoginView.vue` сверстана и работает независимо от `AppLayout`.
2. Реализована логика выбора роли и сохранения fake-сессии.
3. При авторизации происходит успешный редирект в защищенную зону.
4. При обновлении страницы (F5) сессия сохраняется при наличии в `localStorage` токена, роли и **валидного JSON `auth.userInfo`**.
