# Тикет 43: Реализация модуля «Обработка спецификаций»: прием и парсинг CSV-файла

## Описание

Согласно спецификации (п. 2.1.3.3 и 2.3.2), менеджер загружает спецификацию проекта в формате CSV. Серверное приложение должно принять этот файл через `multipart/form-data`, прочитать его содержимое и преобразовать в массив JavaScript-объектов для дальнейшей обработки. Система должна обрабатывать до 2000 строк менее чем за 3 секунды.

## Технические требования

- **Зависимости:**
    - Установить `multer` (для обработки `multipart/form-data`). Использовать `multer.memoryStorage()`, так как файлы небольшие и сохранять их на диск нет необходимости.
    - Установить `csv-parser` или `fast-csv` (для потокового и быстрого парсинга строк).
- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - Создать роут `POST /api/orders/:id/specification`.
    - Подключить middleware авторизации и ролевую модель (только 'Менеджер' или 'Руководитель').
    - Добавить middleware `upload.single('file')`.
- **Сервис (`src/services/SpecificationParser.ts`):**
    - Написать функцию потокового чтения буфера загруженного файла (`req.file.buffer`).
    - Извлечь из каждой строки нужные колонки: `Артикул` (соответствует `article` в БД) и `Количество` (`required_quantity`).

## Критерии приемки (DoD)

1. Эндпоинт загрузки файла создан и защищен.
2. Файл успешно принимается из form-data (в памяти).
3. Парсинг происходит через Stream API (для соблюдения требования к скорости обработки 2000+ строк).
4. На выходе из парсера формируется корректный массив объектов: `[{ article: '...', quantity: 10 }, ...]`.

# Тикет 44: Реализация логики валидации загружаемого CSV-файла (возврат 400 Bad Request при ошибках)

## Описание

Нефункциональное требование (п. 1.8.4.3) жестко регламентирует: система должна корректно обрабатывать исключения при загрузке файлов некорректного формата, возвращая HTTP-статус 400 (Bad Request) без аварийной остановки серверного процесса. Ошибки в загруженном файле не должны "ронять" бэкенд.

## Технические требования

- **Модуль валидации (`src/services/SpecificationParser.ts`):**
    - Проверить наличие `req.file`. Если файл не передан — выбросить `AppError(400, 'Файл не найден')`.
    - Проверить Mime-Type (должен быть `text/csv` или приложение должно пытаться парсить его как CSV и ловить ошибку формата).
    - **Проверка заголовков (Headers):** Убедиться, что в первой строке CSV присутствуют обязательные колонки: «Артикул» и «Количество».
    - **Проверка типов данных:** При потоковом чтении проверять, что значение в колонке «Количество» конвертируется в валидное положительное число (`DECIMAL`). Если попадается `NaN` или `null` — прерывать поток и выбрасывать ошибку.
- **Интеграция с Error Handler:**
    - Все выброшенные исключения должны корректно перехватываться глобальным middleware из Тикета 36.

## Критерии приемки (DoD)

1. При попытке загрузить файл без обязательных колонок («Артикул», «Количество») сервер моментально возвращает HTTP 400 с поясняющим сообщением.
2. При наличии нечисловых данных в колонке «Количество» парсинг прерывается с ошибкой 400.
3. Отправка поврежденных или бинарных файлов (.exe, .png) под видом .csv не приводит к падению сервера (процесс Node.js продолжает работу).

# Тикет 45: Реализация «Сервиса калькуляции»: извлечение базовых цен номенклатуры из БД

## Описание

Для расчета итоговой стоимости проекта нам недостаточно только CSV-файла. В файле указаны лишь Артикул и требуемое Количество. Сервер должен обратиться к складскому справочнику (таблица `materials`), чтобы получить актуальную закупочную/базовую стоимость (`base_cost`) и внутренний системный ID (`material_id`) для каждого извлеченного артикула.

## Технические требования

- **Сервис (`src/services/PricingService.ts`):**
    - Принимает массив, полученный из парсера (Тикет 43): `[{ article: 'A-123', quantity: 5 }, ...]`.
    - Извлекает массив всех уникальных артикулов.
- **Оптимизированный запрос к БД:**
    - Сформировать единый SQL-запрос с оператором `IN`:
      `SELECT id, article, base_cost FROM materials WHERE article IN (?, ?, ...)`
    - _Запрещено_ делать запрос к БД в цикле для каждой строки из CSV (это убьет производительность при 2000 строк).
- **Слияние данных (Маппинг):**
    - Сопоставить ответ базы данных с исходным массивом из CSV по ключу `article`.
    - Сформировать новый массив объектов, содержащий: `material_id`, `required_quantity`, `base_cost`.
- **Обработка отсутствующих деталей:**
    - Если какой-либо артикул из CSV не найден в таблице `materials`, выбросить `AppError(400, 'Артикул [ARTICLE_NAME] не найден в складской номенклатуре')`.

## Критерии приемки (DoD)

1. Написан класс/модуль `PricingService`.
2. Реализовано извлечение цен из базы данных одним SQL-запросом (оператор `IN`).
3. Метод корректно "склеивает" загруженные объемы (`quantity`) с базовой ценой (`base_cost`) из БД.
4. При наличии в CSV несуществующего артикула операция прерывается и пользователю возвращается информативная ошибка (HTTP 400).
