# Тикет 52: Реализация «Модуля складского учета»: получение текущих остатков из `materials`

## Описание

Для работы Кладовщика (Фронтенд-тикет 16) и Закупщика (Фронтенд-тикет 19) сервер должен предоставлять API для получения актуальной информации по всей складской номенклатуре. Этот эндпоинт будет возвращать справочник материалов с их текущим фактическим остатком на полках.

## Технические требования

- **Маршрутизация (`src/routes/inventoryRoutes.ts`):**
    - Создать роут `GET /api/inventory` (или `GET /api/materials`).
    - Защитить через `authMiddleware` (разрешить роли: 'Кладовщик', 'Закупщик', 'Руководитель', 'Менеджер').
- **Контроллер и Служба (`src/controllers/InventoryController.ts`):**
    - Реализовать запрос: `SELECT id, article, name, base_cost, current_stock FROM materials ORDER BY name ASC`.
    - **Поддержка фильтрации (Опционально, для поиска на фронте):** Добавить поддержку query-параметров `?search=...` для поиска совпадений по `article` или `name` с помощью оператора `LIKE '%...%'`.
- **Формат ответа:**
    - Возвращать данные в стандартном формате: `{ status: "success", data: [...] }`. Обратите внимание, что поля `current_stock` и `base_cost` будут возвращаться в виде строк (из-за типа DECIMAL в MySQL), поэтому при необходимости привести их к числам на уровне API перед отправкой, либо обрабатывать на фронтенде.

## Критерии приемки (DoD)

1. Создан эндпоинт `GET /api/inventory`.
2. Роут защищен ролевой моделью.
3. API возвращает полный и корректный список материалов с их остатками.
4. Опциональный поиск по названию или артикулу работает и возвращает отфильтрованный массив.

# Тикет 53: Реализация автоматического резервирования и списания материалов при переходе заказа «В производство»

## Описание

Ключевая бизнес-логика системы (согласно п. 2.3.5 спецификации). Когда заказ переходит в активную фазу производства, система должна автоматически изъять необходимые материалы со склада, опираясь на ранее сохраненную спецификацию. Это действие исключает ручной учет и гарантирует, что детали не будут "проданы" дважды.

## Технические требования

- **Интеграция с `StatusManager` (Доработка Тикета 48/50):**
    - В сервисе `StatusManager` перехватывать событие: `if (newStatus === 'В производстве')`.
- **Логика списания (`src/services/InventoryService.ts`):**
    - В рамках существующей транзакции сделать `SELECT material_id, required_quantity FROM specification_items WHERE order_id = ?`.
    - Если спецификация пуста — пропустить шаг (или залогировать предупреждение).
    - Для каждой позиции выполнить вычитание:
      `UPDATE materials SET current_stock = current_stock - ? WHERE id = ?`.
    - _Важное бизнес-правило:_ Списание **может** уводить `current_stock` в отрицательные значения. Это нормально, так формируется «виртуальный дефицит» для Закупщика. Транзакция не должна прерываться при уходе в минус.
- **Оптимизация (Опционально):**
    - Для ускорения работы вместо множества `UPDATE` в цикле можно использовать SQL-конструкцию `INSERT INTO materials (id, current_stock) VALUES ... ON DUPLICATE KEY UPDATE current_stock = current_stock - VALUES(current_stock)` (или аналогичный пакетный `CASE` update).

## Критерии приемки (DoD)

1. При смене статуса заказа на «В производстве» автоматически запускается процесс списания комплектующих.
2. Объемы списываются строго в соответствии с таблицей `specification_items` для данного `order_id`.
3. Система успешно списывает остатки в минус (создает дефицит), не выбрасывая SQL-ошибок.
4. Вся операция происходит в рамках единой транзакции (вместе со сменой статуса и генерацией сменных заданий).

# Тикет 54: Реализация записи логов движения ТМЦ в таблицу `inventory_transactions`

## Описание

Любое изменение складских остатков (в данном случае — автоматическое списание из Тикета 53) должно быть строго зафиксировано в журнале аудита (`inventory_transactions`). Это необходимо для работы экрана "Истории транзакций" (Фронтенд-тикет 18) и для финансового контроля.

## Технические требования

- **Интеграция с логикой списания (`src/services/InventoryService.ts`):**
    - Расширить метод, реализованный в Тикете 53.
- **Запись логов (SQL):**
    - После (или вместе) выполнения `UPDATE materials` для каждой списываемой детали, необходимо выполнить:
      `INSERT INTO inventory_transactions (material_id, order_id, tx_type, quantity_change) VALUES (?, ?, ?, ?)`
    - `tx_type` установить в константу: `'Списание'` (или `'Резерв'`).
    - `quantity_change` должен быть **отрицательным** числом (например, если списали 5 петель, в лог идет `-5.000`). Значение передавать как `-required_quantity`.
- **Транзакционность:**
    - Обязательно включить эту вставку в общую SQL-транзакцию переключения статуса.

## Критерии приемки (DoD)

1. При списании материалов автоматически генерируются записи в `inventory_transactions`.
2. Логи имеют правильный знак минуса в колонке `quantity_change`.
3. Поле `order_id` в логе корректно заполнено (показывает, под какой именно заказ ушел материал).
4. Процесс записи логов не ломает общую транзакцию и завершается за приемлемое время.
