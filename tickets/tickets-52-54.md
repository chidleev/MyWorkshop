# Тикет 52: Модуль складского учёта: `GET /api/inventory`, создание номенклатуры, физический остаток и резерв

## Описание

Для кладовщика, менеджера, закупщика и руководителя API возвращает справочник материалов с **вычисляемым физическим остатком** и суммой **резерва** по всем заказам. Отдельный маршрут создаёт новую позицию номенклатуры (для ролей, которым это разрешено).

## Технические требования

- **Маршрутизация (`src/routes/inventoryRoutes.ts`):**
    - `POST /api/inventory/materials` — `Менеджер`, `Руководитель`, `Закупщик`: тело `{ article, name, base_cost }` (zod), вставка в `materials` с дефолтным снимком остатка; конфликт артикула → 409.
    - `GET /api/inventory` — `Кладовщик`, `Закупщик`, `Руководитель`, `Менеджер`.
- **Вычисления в SQL (`InventoryController.getInventory`):**
    - Константы типов движений: `INV_TX` в `src/constants/inventoryTransactions.ts` — значения **`Приход`**, **`Резерв`**, **`Списание`** (строго эти строки в `tx_type`).
    - `physical_stock` = выражение `physicalStockExpression('m')`: `stock_snapshot` + сумма `quantity_change` по строкам `inventory_transactions` с `tx_type = 'Списание'` и `tx_date > materials.stock_snapshot_at`.
    - `reserved_change` = сумма `quantity_change` по всем строкам с `tx_type = 'Резерв'` для материала (отрицательное число или ноль).
- **Поиск:** query `?search=` — фильтр `article LIKE` OR `name LIKE` (подстрока).
- **Ответ:** `{ status: "success", data: [{ id, article, name, base_cost, physical_stock, reserved_change }, ...] }` (decimal-поля могут приходить как строки из драйвера).

## Критерии приемки (DoD)

1. `GET /api/inventory` отражает модель «снимок + движения», а не устаревшее поле `current_stock`.
2. Поиск по `search` работает.
3. `POST /api/inventory/materials` создаёт запись и отклоняет дубликат артикула.
4. Роли на маршрутах соответствуют `inventoryRoutes.ts`.

# Тикет 53: Резерв при «В производстве» и конверсия в списание при старте задачи (см. также Тикет 50)

## Описание

Перенос фактического списания с момента перевода заказа в производство на момент начала работ по задаче цеха описан в **Тикете 50** (`syncReserveForOrder`, `convertReserveToConsumptionForOrder`, проверка отрицательного физического остатка). Тикет 53 в текущей архитектуре **не** предполагает `UPDATE materials` при смене стадии заказа.

## Технические требования

- При `newStatus === 'В производстве'` в транзакции `StatusManager` вызывается только `InventoryService.syncReserveForOrder`.
- Физический остаток в БД меняется за счёт **приходов** (обновление `stock_snapshot` / `stock_snapshot_at`) и записей **`Списание`** с датой после снимка; резерв лишь уменьшает «свободно к планированию», не изменяя `stock_snapshot` напрямую.

## Критерии приемки (DoD)

1. Нет прямого вычитания из `materials` при `PATCH /api/orders/:id/status` на `'В производстве'`.
2. Резерв по спецификации согласован с Тикетом 50 и тестами `inventoryReserveAssert` (если есть в репозитории).

# Тикет 54: Журнал `inventory_transactions`: приход, резерв, списание

## Описание

Все движения ТМЦ отражаются в `inventory_transactions`. Типы и знак `quantity_change` должны быть согласованы с расчётом остатков и отчётом дефицита.

## Технические требования

- **Приход (`POST /api/inventory/incoming`):** `INSERT ... tx_type = Приход`, `quantity_change > 0`, `order_id = NULL`; параллельно `UPDATE materials SET stock_snapshot = stock_snapshot + ?, stock_snapshot_at = CURRENT_TIMESTAMP(3)`.
- **Резерв:** создаётся/обновляется `InventoryService.syncReserveForOrder` — одна строка на пару заказ–материал, `quantity_change` отрицательное, `tx_type = Резерв`.
- **Списание:** конверсия из резерва (`UPDATE` типа и `tx_date`) либо прямое списание в будущих сценариях — в текущей версии основной путь конверсии при старте задачи.
- Все вставки/обновления в рамках транзакций соответствующих операций.

## Критерии приемки (DoD)

1. В журнале различаются `Приход`, `Резерв`, `Списание` по полю `tx_type`.
2. Знак `quantity_change` согласован с операцией (плюс для прихода, минус для резерва).
3. Экран истории (Тикет 18 / 57) может фильтровать по этим значениям.
