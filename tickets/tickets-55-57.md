# Тикет 55: Реализация эндпоинта фиксации прихода фурнитуры и обновления баланса

## Описание

Этот эндпоинт обеспечивает работу Кладовщика по пополнению виртуальных остатков (Фронтенд-тикет 17). Когда в мастерскую приезжает новая партия фурнитуры или плитных материалов от поставщика, система должна увеличить фактический остаток и зафиксировать это событие в журнале складских транзакций.

## Технические требования

- **Маршрутизация (`src/routes/inventoryRoutes.ts`):**
    - Создать роут `POST /api/inventory/incoming`.
    - Защитить эндпоинт через `authMiddleware` (разрешить роли: 'Кладовщик', 'Закупщик', 'Руководитель').
- **Контроллер и Служба:**
    - Принимает JSON-тело: `{ "material_id": 123, "quantity": 50.500 }`.
    - Провести валидацию: `quantity` должно быть строго положительным числом.
- **Логика БД (SQL Транзакция):**
    - Инициализировать транзакцию.
    - Выполнить пополнение: `UPDATE materials SET current_stock = current_stock + ? WHERE id = ?`.
    - Сразу же записать лог операции: `INSERT INTO inventory_transactions (material_id, order_id, tx_type, quantity_change) VALUES (?, NULL, 'Приход', ?)`.
    - _Важно:_ Поле `order_id` остается `NULL`, так как приход обычно осуществляется на склад в целом, а не под конкретный заказ. Поле `quantity_change` должно быть положительным.
    - Закоммитить транзакцию.

## Критерии приемки (DoD)

1. Эндпоинт `POST /api/inventory/incoming` создан и защищен.
2. При вызове эндпоинта поле `current_stock` в таблице `materials` корректно увеличивается.
3. В таблице `inventory_transactions` создается запись с типом `Приход` и положительным значением количества.
4. Обновление баланса и запись лога происходят в одной SQL-транзакции.
5. Некорректные данные (например, отрицательный приход) отклоняются с ошибкой 400 Bad Request.

# Тикет 56: Реализация эндпоинта генерации отчета о дефиците комплектующих (отрицательные остатки)

## Описание

Данный эндпоинт является источником данных для рабочего пространства Закупщика (Фронтенд-тикет 19). Из-за автоматического списания под новые сделки, некоторые материалы могут уходить в виртуальный «минус». Сервер должен вернуть список таких позиций для своевременного формирования заказов поставщикам.

## Технические требования

- **Маршрутизация (`src/routes/inventoryRoutes.ts`):**
    - Создать роут `GET /api/inventory/deficit`.
    - Защитить через `authMiddleware` (разрешить роли: 'Закупщик', 'Руководитель').
- **Контроллер и БД (`src/controllers/InventoryController.ts`):**
    - Сформировать простой и быстрый SQL-запрос:
      `SELECT id, article, name, base_cost, current_stock FROM materials WHERE current_stock < 0 ORDER BY current_stock ASC`.
    - _Опционально:_ На уровне SQL или контроллера можно сразу вычислить сумму закупки по каждой позиции (например, `ABS(current_stock) * base_cost AS total_deficit_cost`).
- **Формат ответа:**
    - JSON-объект с массивом дефицитных материалов и (желательно) подсчитанным итоговым бюджетом на закупку (сумма всех `total_deficit_cost`).

## Критерии приемки (DoD)

1. Создан эндпоинт `GET /api/inventory/deficit`.
2. Роут возвращает только те материалы, у которых `current_stock` строго меньше нуля.
3. Данные отсортированы по размеру дефицита (от самых недостающих к менее недостающим).
4. Запрос выполняется быстро благодаря индексам (если есть) и простой выборке.

# Тикет 57: Реализация эндпоинта детализации движения комплектующих по сделкам (мониторинг резервирования)

## Описание

Эндпоинт предоставляет данные для экрана "Истории транзакций" Кладовщика (Фронтенд-тикет 18). Он должен возвращать хронологический список всех складских операций (приходы, резервы, списания), объединяя данные логов с информацией о самом материале.

## Технические требования

- **Маршрутизация (`src/routes/inventoryRoutes.ts`):**
    - Создать роут `GET /api/inventory/transactions`.
    - Защитить через `authMiddleware` (разрешить роли: 'Кладовщик', 'Руководитель').
- **Контроллер и БД:**
    - Написать SQL-запрос с `JOIN`:
      `SELECT t.id, t.tx_date, t.tx_type, t.quantity_change, t.order_id, m.article, m.name FROM inventory_transactions t JOIN materials m ON t.material_id = m.id ORDER BY t.tx_date DESC`.
- **Реализация фильтрации (Query-параметры):**
    - Поддержать фильтр `?tx_type=...` (например, только `Приход` или только `Списание`).
    - Поддержать фильтр `?order_id=...` для просмотра истории по конкретной сделке.
- **Пагинация (Опционально, но рекомендуется):**
    - Так как лог со временем станет огромным, добавить параметры `limit` и `offset` (по умолчанию возвращать последние 100-200 записей).

## Критерии приемки (DoD)

1. Эндпоинт `GET /api/inventory/transactions` создан и работает.
2. В ответе присутствуют объединенные данные: информация о транзакции + артикул и название детали.
3. Записи отсортированы в хронологическом порядке (сначала новые).
4. Корректно работают query-фильтры по типу операции и ID заказа.
