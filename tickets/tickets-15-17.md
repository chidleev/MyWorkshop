# Тикет 15: Создание компонента формы для прикрепления фотоотчета и кнопки «Завершить»

## Описание

Завершающим шагом работы монтажника на объекте является фиксация результата. Монтажник должен сфотографировать собранную мебель (шкаф или кухню) и прикрепить фото в систему. Это действие служит бизнес-триггером: заказ переводится в финальную стадию «Завершено»; бэкенд автоматически формирует PDF чека и акта, пишет ссылки в `media_files` и отправляет клиенту письмо со ссылками (Тикеты 62–63, `NotificationService` / `ClosingDocumentsService`).

## Технические требования

- **Компонент (`src/components/Installer/PhotoReportUploader.vue`):**
    - Встроить компонент на страницу деталей монтажа (из Тикета 14).
    - Использовать `<input type="file" accept="image/*" capture="environment" />`, чтобы на мобильных устройствах можно было сразу открыть камеру.
- **Предпросмотр (Preview):**
    - После выбора фотографии показывать её миниатюру на экране перед окончательной отправкой.
    - Добавить возможность удалить выбранное фото и переснять.
- **Действие завершения:**
    - Цепочка: `uploadMedia` из `src/api/media.ts`, затем `updateOrderStatus(orderId, 'Завершен')` из `src/api/orders.ts`; индикатор загрузки на время всей цепочки.
- **UX/UI нюансы:**
    - После "успешной" отправки показывать полноэкранное зеленое уведомление: «Объект успешно сдан!» и возвращать пользователя к списку выездов.

## Критерии приемки (DoD)

1. Реализован загрузчик фотографий с поддержкой прямой съемки с камеры устройства.
2. Корректно работает предпросмотр выбранного снимка.
3. Кнопка «Завершить монтаж» заблокирована до момента прикрепления фото.
4. При успешном сабмите статус заказа обновляется, и монтажник видит уведомление об успешной сдаче объекта.

# Тикет 16: Разработка цифровой панели физических остатков материалов (Кладовщик)

## Описание

Рабочий экран для Кладовщика. Ему необходима цифровая панель, которая в режиме реального времени показывает физическое наличие плитных материалов и фурнитуры на складских полках. Этот экран заменяет бумажные журналы и позволяет быстро оценить, что есть в наличии.

## Технические требования

- **Страница (`src/views/Storekeeper/InventoryView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Кладовщик'`.
    - Таблица остатков на **`AppDataTable`**: колонки по полям API (артикул, наименование, остаток на базе **`physical_stock`**; при необходимости отдельно отображать влияние **`reserved_change`** — например переключатель «Учитывать резервы» и вычисляемая колонка «остаток к отображению»).
- **Стейт / API:**
    - `fetchInventory({ search })` → `GET /api/inventory` с query `search` при необходимости.
- **Функционал фильтрации:**
    - Добавить строку поиска по Артикулу и Наименованию для быстрого нахождения нужной детали.
- **SPA / UX (без «ложной перезагрузки» страницы):**
    - Разделить **первичную загрузку** и **фоновое обновление** после операций (в т.ч. приход из Тикета 17): при повторном `fetchInventory` не скрывать всю таблицу под блоком «Загрузка остатков…» — использовать режим «мягкого» обновления (например `loadInventory({ soft: true })`, отдельный флаг `isRefreshing`, короткий текст «Обновление остатков…»). Полноэкранный индикатор загрузки допустим только при первом открытии экрана или когда данных ещё нет.

## Критерии приемки (DoD)

1. Создан защищенный роут `/storekeeper/inventory`. При логине под Кладовщиком происходит редирект на этот экран.
2. Таблица строится из ответа `GET /api/inventory`.
3. Поиск по артикулу и названию фильтрует таблицу без задержек.
4. Визуальное выделение строк, где отображаемый остаток (в выбранном режиме с/без резервов) меньше или равен нулю.
5. После успешного прихода (Тикет 17) таблица обновляется **без полного исчезновения** списка из-за глобального `isLoading` (мягкое обновление, как в технических требованиях).

# Тикет 17: Реализация формы фиксации прихода новой партии фурнитуры

## Описание

Кладовщик должен своевременно вносить данные о поступивших от контрагентов (поставщиков) материалах. Это действие пополняет виртуальные остатки системы, открывая возможность запуска новых проектов в производство (так как устраняется дефицит).

## Технические требования

- **Компонент (`src/components/Storekeeper/IncomingStockModal.vue`):**
    - Модальное окно (или side-panel), вызываемое со страницы `InventoryView` по кнопке «Оформить приход».
    - **Разметка формы:** поля обёрнуты в `<form @submit.prevent="...">`; кнопка подтверждения — `type="submit"` (или эквивалент с явным `preventDefault` на submit), чтобы **Enter** в поле количества вызывал тот же сценарий сохранения и **не** приводил к нативной отправке документа / перезагрузке вкладки.
    - **Поля формы:**
        - Выбор номенклатуры (Select или автокомплит по `article`/`name` из существующего списка материалов).
        - Количество поступившего материала (`quantity_change`) — _обязательное числовое поле (DECIMAL/Float)_.
- **Логика работы:**
    - При сохранении вызывать `registerIncomingStock` из `src/api/inventory.ts` (`POST /api/inventory/incoming`).
    - Окно автоматически закрывается, форма сбрасывается.
- **Обновление родителя (`InventoryView`):**
    - После успеха вызывать повторную загрузку остатков в режиме **мягкого обновления** (см. Тикет 16: `loadInventory({ soft: true })` или тот же UX), чтобы не скрывать таблицу глобальным индикатором первичной загрузки.
- **Уведомления:**
    - Показать `showSuccess` из `src/utils/notification.ts`: «Остатки успешно пополнены».

## Критерии приемки (DoD)

1. Сверстано модальное окно оформления прихода.
2. Выпадающий список номенклатуры корректно подтягивает доступные материалы.
3. Валидация не пропускает пустые или отрицательные значения количества.
4. После успеха родитель заново получает данные с API (`fetchInventory` / обёртка над ним), таблица обновляется **без эффекта полной перезагрузки SPA** (мягкое обновление).
5. Отправка с клавиатуры (Enter) в модалке не вызывает нативной перезагрузки страницы.
