# Тикет 67: Интеграция спецификации: CSV, preview/commit, чтение, наценка

## Описание

Фронтенд использует полный набор методов `src/api/orders.ts`: загрузка CSV, предпросмотр и коммит JSON-строк, чтение сохранённой спецификации, изменение наценки — в соответствии с Тикетами 46–47 и маршрутами `orderRoutes`.

## Технические требования

- **Методы API:**
    - `uploadSpecification(orderId, file)` → `POST /api/orders/:id/specification` (`multipart/form-data`, поле `file`).
    - `previewOrderSpecification(orderId, rows)` → `POST /api/orders/:id/specification/preview` с телом `{ rows: [{ article, quantity }, ...] }`.
    - `commitOrderSpecification(orderId, rows)` → `POST /api/orders/:id/specification/commit`.
    - `fetchOrderSpecification(orderId)` → `GET /api/orders/:id/specification`.
    - `patchOrderPricingMarkup(orderId, { pricing_markup_mode, pricing_markup_value })` → `PATCH /api/orders/:id/pricing-markup`.
- **UI (`SpecificationUploader.vue`, `OrderDetailView.vue`):**
    - Axios при `FormData` не должен принудительно перетирать `Content-Type` (или явно выставить `multipart/form-data` — как в текущей реализации).
    - Ответы обрабатываются как обёртка axios-interceptor'а (`response` уже `data`): использовать поля `items` / унифицированную структуру `SpecificationMutationResponse` / `OrderSpecificationResponse` из типов в `orders.ts`.
    - Блокировка двойной отправки, toast при 400 (Тикет 24).

## Критерии приемки (DoD)

1. Все перечисленные методы используются согласованно с бэкендом.
2. После операций таблица и блок «итого / наценка» отражают серверное состояние.
3. Ошибки наценки или стадии ≠ `Новый` отображаются пользователю.

# Тикет 68: Интеграция переключения статусов: вызов API из карточки задания мастера цеха

## Описание

Оживление табло мастера (Тикеты 11–12): смена **статуса сменного задания** идёт через **`PATCH /api/workshop/tasks/:id/status`** (Тикет 59), а не через `PATCH /api/orders/:id/status` (последний используется для стадий заказа в других ролях/сценариях).

## Технические требования

- **API-сервис (`src/api/workshop.ts`):**
    - `fetchTasks(status?)` → `GET /api/workshop/tasks`.
    - `updateWorkshopTaskStatus(taskId, 'new' | 'in_progress' | 'done')` → маппинг на русские значения `Ожидает` / `В работе` / `Завершен` в теле `PATCH /api/workshop/tasks/:id/status`.
- **Интеграция в `ShiftTaskCard.vue`:** при клике — спиннер/дизейбл кнопки, вызов `updateWorkshopTaskStatus`, по успеху родитель вызывает `fetchTasks` (emit или колбэк). Ошибки — toast и сброс состояния кнопки.

## Критерии приемки (DoD)

1. Кнопки мастера вызывают **`PATCH /api/workshop/tasks/:id/status`**, а не смену стадии заказа напрямую.
2. Интерфейс блокируется на время сетевого запроса, предотвращая дублирование кликов.
3. При успешном ответе UI реактивно обновляется (задача перемещается в соответствующую колонку/меняет цвет).
4. Ошибки API корректно перехватываются.

# Тикет 69: Интеграция функционала монтажника: загрузка фотоотчета на сервер и триггер завершения сделки

## Описание

Связывание флоу выездного монтажника (Фронтенд-тикет 15) с серверным API (Бэкенд-тикет 51). Монтажник делает фото собранной мебели и нажимает «Завершить». Фронтенд должен загрузить медиафайл и инициировать перевод сделки в финальный статус.

## Технические требования

- **API-сервисы (`src/api/media.ts` и `src/api/orders.ts`):**
    - Создать метод `uploadMedia(orderId: number, file: File, type: string = 'Фото')` для выполнения `POST /api/orders/${orderId}/media` (`multipart/form-data`).
- **Интеграция в компонент (`src/components/Installer/PhotoReportUploader.vue`):**
    - В функции обработчике кнопки «Завершить монтаж» реализовать цепочку промисов (или `async/await`):
        1. Включить глобальный лоадер.
        2. Вызвать `uploadMedia()` с прикрепленным фото.
        3. Если загрузка фото успешна, вызвать `updateOrderStatus(orderId, 'Завершен')`.
        4. Выключить лоадер.
    - При успехе показать сообщение «Объект успешно сдан» и выполнить `router.push('/installer/deployments')` для возврата к списку маршрутов.
- **Обработка сбоев:**
    - Если загрузка фото упала (например, из-за плохой связи на объекте), прервать цепочку, не менять статус заказа и вывести ошибку.

## Критерии приемки (DoD)

1. Фотография успешно отправляется на сервер и сохраняется в таблице `media_files` (можно проверить через БД или панель разработчика).
2. Сразу после загрузки фото статус заказа меняется на «Завершен».
3. Весь процесс сопровождается понятным для пользователя индикатором загрузки.
4. При сбоях интернета на телефоне монтажника система корректно показывает ошибку и позволяет повторить отправку.
