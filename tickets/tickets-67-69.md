# Тикет 67: Интеграция импорта CSV: отправка файла на бэкенд, обработка ответа, вывод итоговой цены на фронт

## Описание

Необходимо связать загрузчик спецификаций (Фронтенд-тикеты 9 и 10) с реализованным серверным эндпоинтом парсинга и калькуляции (Бэкенд-тикеты 43-47). Пользовательский интерфейс должен отправить CSV-файл на сервер, дождаться расчета и реактивно отобразить загруженные детали и обновленную итоговую стоимость заказа.

## Технические требования

- **API-сервис (`src/api/orders.ts`):**
    - Добавить метод `uploadSpecification(orderId: number, file: File)`.
    - Метод должен формировать `FormData`, добавлять файл (`formData.append('file', file)`) и отправлять `POST /api/orders/${orderId}/specification`.
    - Установить правильные заголовки: `headers: { 'Content-Type': 'multipart/form-data' }` (хотя Axios обычно делает это сам при передаче FormData).
- **Интеграция в компонент (`src/components/Orders/SpecificationUploader.vue` и `OrderDetailView.vue`):**
    - Заменить `setTimeout` (mock-логику) на реальный вызов `uploadSpecification()`.
    - Во время запроса блокировать кнопку загрузки и показывать спиннер.
    - Обработать успешный ответ (HTTP 200): извлечь из `response.data` массив деталей и значение `total_cost`, обновить локальный стейт страницы для перерисовки таблицы и итоговой суммы.
    - Обработать ошибку (HTTP 400): перехватить ошибку парсинга и вывести ее через глобальный компонент уведомлений (Тикет 24).

## Критерии приемки (DoD)

1. Компонент загрузки успешно отправляет реальный CSV-файл на сервер.
2. При успешном ответе таблица спецификации и итоговая цена (`total_cost`) мгновенно обновляются на экране без перезагрузки страницы.
3. Состояние загрузки (isUploading) корректно блокирует интерфейс от двойной отправки.
4. Ошибки неверного формата файла корректно отлавливаются и выводятся в виде красного Toast-уведомления.

# Тикет 68: Интеграция переключения статусов: вызов API из карточки задания мастера цеха

## Описание

Оживление интерфейса производственного цеха (Фронтенд-тикеты 11 и 12). Интерактивные кнопки на планшете мастера ("Взять в работу", "Завершить") должны отправлять сетевые запросы на сервер (Бэкенд-тикет 48) для реального изменения статусов в базе данных.

## Технические требования

- **API-сервис (`src/api/status.ts` или `src/api/orders.ts`):**
    - Добавить метод `updateOrderStatus(orderId: number, status: string)` для выполнения `PATCH /api/orders/${orderId}/status`.
    - _Дополнительно (опционально):_ Если в архитектуре предусмотрен отдельный эндпоинт для обновления статуса конкретной _задачи_ (task), добавить метод `updateTaskStatus(taskId: number, status: string)`.
- **Интеграция в карточку (`src/components/Workshop/ShiftTaskCard.vue`):**
    - При клике на кнопку смены статуса переводить карточку в состояние загрузки (показывать локальный спиннер внутри кнопки, дизейблить её).
    - Выполнить API-запрос.
    - После успешного ответа сервера сэмитить событие для родительского компонента, чтобы тот заново запросил актуальный список заданий (`fetchTasks`), либо локально обновить статус в стейте (Optimistic UI update).
- **Обработка ошибок:**
    - Если запрос упал, вернуть кнопку в исходное состояние и показать Toast с ошибкой.

## Критерии приемки (DoD)

1. При клике на кнопки в карточке мастера отправляются реальные PATCH-запросы на бэкенд.
2. Интерфейс блокируется на время сетевого запроса, предотвращая дублирование кликов.
3. При успешном ответе UI реактивно обновляется (задача перемещается в соответствующую колонку/меняет цвет).
4. Ошибки API корректно перехватываются.

# Тикет 69: Интеграция функционала монтажника: загрузка фотоотчета на сервер и триггер завершения сделки

## Описание

Связывание флоу выездного монтажника (Фронтенд-тикет 15) с серверным API (Бэкенд-тикет 51). Монтажник делает фото собранной мебели и нажимает «Завершить». Фронтенд должен загрузить медиафайл и инициировать перевод сделки в финальный статус.

## Технические требования

- **API-сервисы (`src/api/media.ts` и `src/api/orders.ts`):**
    - Создать метод `uploadMedia(orderId: number, file: File, type: string = 'Фото')` для выполнения `POST /api/orders/${orderId}/media` (`multipart/form-data`).
- **Интеграция в компонент (`src/components/Installer/PhotoReportUploader.vue`):**
    - В функции обработчике кнопки «Завершить монтаж» реализовать цепочку промисов (или `async/await`):
        1. Включить глобальный лоадер.
        2. Вызвать `uploadMedia()` с прикрепленным фото.
        3. Если загрузка фото успешна, вызвать `updateOrderStatus(orderId, 'Завершен')`.
        4. Выключить лоадер.
    - При успехе показать сообщение «Объект успешно сдан» и выполнить `router.push('/installer/deployments')` для возврата к списку маршрутов.
- **Обработка сбоев:**
    - Если загрузка фото упала (например, из-за плохой связи на объекте), прервать цепочку, не менять статус заказа и вывести ошибку.

## Критерии приемки (DoD)

1. Фотография успешно отправляется на сервер и сохраняется в таблице `media_files` (можно проверить через БД или панель разработчика).
2. Сразу после загрузки фото статус заказа меняется на «Завершен».
3. Весь процесс сопровождается понятным для пользователя индикатором загрузки.
4. При сбоях интернета на телефоне монтажника система корректно показывает ошибку и позволяет повторить отправку.
