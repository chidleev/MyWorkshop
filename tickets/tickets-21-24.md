# Тикет 21: Создание дашборда контроля загрузки производственных линий и бригад

## Описание

Руководитель должен понимать, где на производстве образуются «узкие горлышки» (bottlenecks). На дашборде он отслеживает распределение сменных заданий по рабочим центрам (распил, кромление, присадка, сборка) и видит, где скапливается наибольшее количество невыполненных операций.

## Технические требования

- **Страница (`src/views/Director/WorkloadDashboardView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Визуализация: **нативные** прогресс-бары / проценты по строкам операций (без обязательного `Chart.js`); данные из `fetchWorkload()` после проверки `isServerAvailable` (`src/utils/serverHealth.ts`), как на дефиците/других экранах.
- **Логика отображения данных:**
    - При монтировании вызвать `fetchWorkload()` из `src/api/director.ts` (`GET /api/director/workload`).
    - Ответ: вложенный объект `{ [operation_name]: { [progress_status]: count } }` — построить карточки по операциям; допускается маппинг русских статусов и латинских ключей (`new` / `in_progress` / `done`) для устойчивости.
- **UX/UI нюансы:**
    - Визуально выделять (красная рамка / бейдж «Перегруз») операции, где число задач в статусе «Ожидает» (или `new`) **строго больше** порога `pendingLimit` (в коде базово `2`).

## Критерии приемки (DoD)

1. Создан роут `/director/workload` (только для Руководителя).
2. Дашборд строится из ответа `GET /api/director/workload`, без локального mock-агрегата задач.
3. Прогресс-бары/проценты наглядно показывают загрузку по операциям из реального ответа API.
4. Выполнена визуальная подсветка перегруженных участков (узких мест).

# Тикет 22: Реализация экрана финансового отчета для анализа рентабельности проектов

## Описание

Одной из важнейших целей автоматизации является получение данных о прибыльности каждого шкафа или кухни. Система должна сопоставлять итоговую стоимость сделки (Revenue) с реальными затратами на израсходованную фурнитуру и материалы (COGS), вычисляя чистую маржу.

## Технические требования

- **Страница (`src/views/Director/ProfitabilityReportView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Отображение данных: KPI-карточки сверху + финансовая таблица закрытых сделок на **`AppDataTable`** (статус заказов на бэкенде — только «Завершен»).
- **Колонки таблицы:**
    - Номер заказа и ФИО клиента.
    - Сумма договора (`total_cost` из `orders`).
    - Себестоимость материалов (сумма `base_cost * required_quantity` из спецификации).
    - Валовая прибыль (Сумма договора минус Себестоимость).
    - Рентабельность в процентах (Валовая прибыль / Сумма договора \* 100).
- **Источник данных:**
    - `fetchProfitability()` → `GET /api/director/profitability`; использовать поля `revenue`, `cogs`, `profit`, `margin_percent` из ответа.
- **UI элементы:**
    - Карточки KPI наверху можно посчитать как сумму/среднее по загруженным строкам (как на реализованном экране).

## Критерии приемки (DoD)

1. Роут `/director/profitability` создан и защищен.
2. В таблице отображаются только завершенные сделки.
3. Расчёты в таблице соответствуют данным бэкенда (в т.ч. `margin_percent`).
4. Показатели в процентах и валюте отформатированы (например, `150 000.00 ₽`).

# Тикет 23: Создание виджета формы обратной связи для встраивания на корпоративный сайт (Клиент)

## Описание

Согласно архитектуре (С4, Уровень 2), сторонние площадки или корпоративный сайт могут генерировать новые заказы через API. Необходимо создать изолированный Vue-компонент (виджет), который можно сбилдить отдельно и встроить на любой внешний сайт для сбора первичных лидов (заявок).

## Технические требования

- **Компонент (`src/components/Client/LeadCaptureWidget.vue`):**
    - Изолированный компонент формы без привязки к глобальному AppLayout (демо-роут `/feedback-widget` в `router/index.ts`).
    - **Поля формы:** Имя (`name`, обязательно), **E-mail** (обязательно, проверка формата), Телефон (необязательно; при вводе — нормализация через `src/utils/phone`: `normalizePhoneInput`, `isValidRuPhone`), Комментарий (необязательно).
- **Логика работы:**
    - `computed` `isFormValid`: имя непустое, email валиден, телефон либо пуст, либо проходит `isValidRuPhone`.
    - Сабмит: `createExternalLead` из `src/api/external.ts` → **`POST /api/external/leads`** с телом `{ client_name, client_email, client_phone?, comment? }` (в теле дублируется поле `email` из `client_email` для совместимости с бэкендом).
    - Состояния: `isLoading`, `isSuccess`, `errorMessage`; после успеха сохранить `agreement_number` из ответа.
- **UX/UI нюансы:**
    - После успешной отправки — блок благодарности с номером договора из ответа API.

## Критерии приемки (DoD)

1. Компонент формы сверстан и работает изолированно (в т.ч. на демо-роуте без авторизации).
2. Клиентская валидация: имя, email, опционально телефон по правилам выше.
3. Кнопка отправки блокируется на время реального HTTP-запроса.
4. Успех отображается только после HTTP 201 и валидного тела ответа с `agreement_number`.

# Тикет 24: Разработка глобального компонента обработки и отображения HTTP-ошибок

## Описание

Для качественного UX система должна явно реагировать на ошибки (как клиентские, так и серверные). Согласно нефункциональным требованиям, сервер может возвращать HTTP 400 Bad Request при загрузке некорректных CSV-файлов. Нужен глобальный механизм перехвата таких ошибок и показа их пользователю (Toasts / Snackbars).

## Технические требования

- **Система уведомлений (без отдельной npm-библиотеки toast):**
    - Pinia-стор `src/stores/notifications.ts` (очередь сообщений с типом `success` | `error` | `warning`).
    - Компонент `src/components/GlobalToasts.vue`, подключённый в `App.vue` над `<RouterView />`.
- **Утилита (`src/utils/notification.ts`):**
    - Функции `showSuccess`, `showError`, `showWarning` — пушат событие в стор уведомлений.
- **Интеграция с Axios (Тикет 64):**
    - В `src/api/axios.ts` response-interceptor при 401 выполняет `logout` и редирект на именованный роут `login`; при 403 — `showError` и редирект на `getDefaultRouteForRole`; при 400, 409, 5xx и отсутствии ответа сети — `showError` с текстом из `response.data.message` или запасной строкой.
- **Интеграция с CSV-загрузчиком (Тикет 9):**
    - `SpecificationUploader.vue` при ошибке загрузки/ответа сервера показывает сообщение пользователю (через цепочку ошибки из axios / `showError`).

## Критерии приемки (DoD)

1. `GlobalToasts` отображает уведомления из стора; `App.vue` монтирует его глобально.
2. `showSuccess` / `showError` / `showWarning` доступны через импорт из `src/utils/notification.ts`.
3. Визуально различимы типы сообщений (успех / ошибка / предупреждение).
4. Ошибки API и сценарии CSV отображаются пользователю без падения приложения.
