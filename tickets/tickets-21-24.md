# Тикет 21: Создание дашборда контроля загрузки производственных линий и бригад

## Описание

Руководитель должен понимать, где на производстве образуются «узкие горлышки» (bottlenecks). На дашборде он отслеживает распределение сменных заданий по рабочим центрам (распил, кромление, присадка, сборка) и видит, где скапливается наибольшее количество невыполненных операций.

## Технические требования

- **Страница (`src/views/Director/WorkloadDashboardView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Сверстать дашборд с использованием прогресс-баров или простых диаграмм (рекомендуется использовать легковесную библиотеку, например `Chart.js` (через `vue-chartjs`) или нативные CSS-бары).
- **Логика отображения данных:**
    - Сформировать mock-данные на основе таблицы `tasks` (Сменные задания).
    - Сгруппировать задачи по типу операции (`operation_name`).
    - Для каждой операции показать соотношение статусов: «Ожидают» / «В работе» / «Завершены».
- **UX/UI нюансы:**
    - Визуально выделять (например, красным цветом) те рабочие центры, где количество задач со статусом «Ожидают» превышает допустимый лимит (критическая нагрузка).

## Критерии приемки (DoD)

1. Создан роут `/director/workload` (только для Руководителя).
2. Дашборд корректно рендерит агрегированные данные по производственным линиям.
3. Графики/прогресс-бары наглядно показывают загрузку оборудования и персонала.
4. Выполнена визуальная подсветка перегруженных участков (узких мест).

# Тикет 22: Реализация экрана финансового отчета для анализа рентабельности проектов

## Описание

Одной из важнейших целей автоматизации является получение данных о прибыльности каждого шкафа или кухни. Система должна сопоставлять итоговую стоимость сделки (Revenue) с реальными затратами на израсходованную фурнитуру и материалы (COGS), вычисляя чистую маржу.

## Технические требования

- **Страница (`src/views/Director/ProfitabilityReportView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Отображение данных: Финансовая таблица закрытых сделок (статус "Завершен").
- **Колонки таблицы:**
    - Номер заказа и ФИО клиента.
    - Сумма договора (`total_cost` из `orders`).
    - Себестоимость материалов (сумма `base_cost * required_quantity` из спецификации).
    - Валовая прибыль (Сумма договора минус Себестоимость).
    - Рентабельность в процентах (Валовая прибыль / Сумма договора \* 100).
- **Стейт / Mock-данные:**
    - Написать вспомогательную computed-функцию или геттер в Pinia, которая на лету сводит mock-данные из заказов, спецификаций и справочника материалов для расчета этих колонок.
- **UI элементы:**
    - Отображение итоговых финансовых показателей по всей мастерской (карточки сверху: "Общая выручка", "Общая себестоимость", "Средняя маржинальность").

## Критерии приемки (DoD)

1. Роут `/director/profitability` создан и защищен.
2. В таблице отображаются только завершенные сделки.
3. Все математические расчеты (выручка, себестоимость, маржа) работают корректно на основе mock-данных.
4. Показатели в процентах и валюте отформатированы (например, `150 000.00 ₽`).

# Тикет 23: Создание виджета формы обратной связи для встраивания на корпоративный сайт (Клиент)

## Описание

Согласно архитектуре (С4, Уровень 2), сторонние площадки или корпоративный сайт могут генерировать новые заказы через API. Необходимо создать изолированный Vue-компонент (виджет), который можно сбилдить отдельно и встроить на любой внешний сайт для сбора первичных лидов (заявок).

## Технические требования

- **Компонент (`src/components/Client/LeadCaptureWidget.vue`):**
    - Изолированный компонент формы без привязки к глобальному AppLayout.
    - **Поля формы:** Имя (обязательно), Телефон (обязательно, маска ввода), Комментарий/Пожелание к мебели (текстовое поле).
- **Логика работы:**
    - Валидация заполненности полей.
    - При клике на «Оставить заявку» имитируется POST-запрос на API бэкенда (создание нового заказа).
    - Состояние загрузки (disabled кнопка, спиннер).
- **UX/UI нюансы:**
    - После успешной отправки форма должна заменяться на блок с благодарностью: «Спасибо! Ваша заявка принята. Ожидайте звонка менеджера».

## Критерии приемки (DoD)

1. Компонент формы сверстан и работает абсолютно изолированно.
2. Присутствует базовая клиентская валидация (проверка маски телефона и пустого имени).
3. Состояние кнопки "Отправить" меняется во время имитации сетевого запроса.
4. После "отправки" клиент видит Success Screen.

# Тикет 24: Разработка глобального компонента обработки и отображения HTTP-ошибок

## Описание

Для качественного UX система должна явно реагировать на ошибки (как клиентские, так и серверные). Согласно нефункциональным требованиям, сервер может возвращать HTTP 400 Bad Request при загрузке некорректных CSV-файлов. Нужен глобальный механизм перехвата таких ошибок и показа их пользователю (Toasts / Snackbars).

## Технические требования

- **Система уведомлений:**
    - Установить и настроить легковесную библиотеку для всплывающих уведомлений (например, `vue-toastification` или написать свой менеджер тостов через Pinia и глобальный компонент `GlobalToasts.vue` в корне `App.vue`).
- **Создание сервиса/утилиты (`src/utils/notification.ts`):**
    - Реализовать функции: `showSuccess(msg)`, `showError(msg)`, `showWarning(msg)`.
- **Интеграция с CSV-загрузчиком (рефакторинг Тикета 9):**
    - Внедрить использование функции `showError('Некорректный формат файла. Убедитесь, что загружаете правильный CSV.')` в компонент `SpecificationUploader.vue` при имитации ошибки 400.

## Критерии приемки (DoD)

1. Глобальный компонент нотификаций установлен/создан и прописан в `App.vue`.
2. Методы вызова уведомлений доступны из любой точки приложения (через импорт утилиты или глобальные свойства).
3. Присутствуют стили для успешных, ошибочных и предупреждающих сообщений (цветовое кодирование: зеленый, красный, желтый).
4. Загрузчик CSV-файлов успешно использует новый механизм для вывода сообщения о "поврежденном файле".
