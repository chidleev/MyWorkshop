# Тикет 34: Инициализация проекта Node.js + Express.js

## Описание

Старт разработки серверной части информационной системы. Необходимо инициализировать проект, настроить среду выполнения (Node.js + TypeScript) и базовый веб-сервер на Express.js. Сервер будет выступать единой точкой входа (REST API) для фронтенд-приложения и внешних систем.

## Технические требования

- **Среда выполнения:** Node.js (версия 18.x или выше).
- **Стек:** Express.js, TypeScript.
- **Структура проекта:**
    - Создать базовую слоистую архитектуру папок: `src/controllers`, `src/services`, `src/routes`, `src/middlewares`, `src/config`, `src/utils`.
- **Скрипты (`package.json`):** `dev` — `tsx watch src/server.ts` (точка входа процесса, вызывающая `startServer` из `app.ts`); `build` — `tsc`; `start` — `node dist/app.js` (поле `"main": "dist/app.js"` — для Phusion Passenger и прод-запуска используется скомпилированный **`app.js`**, а не `server.ts`).
- **Конфигурация:**
    - Инициализировать `package.json` и установить базовые зависимости (`express`, `cors`, `dotenv`, `helmet`, `typescript`, `@types/*`, `tsx` для локальной разработки).
    - Настроить `tsconfig.json` (target: ES2022, module: CommonJS/ESNext, strict: true).
    - Создать файл `.env` для переменных окружения (в первую очередь `PORT=3000`).
- **Базовый роут:**
    - Реализовать healthcheck-эндпоинт `GET /api/health` для проверки доступности сервера (возвращает `{ status: 'ok', timestamp: '...' }`).

## Критерии приемки (DoD)

1. Проект успешно инициализирован, скрипты `dev` и `build` прописаны в `package.json`.
2. TypeScript настроен в строгом режиме (`strict: true`).
3. Сервер успешно запускается без ошибок и слушает заданный порт.
4. Эндпоинт `GET /api/health` корректно отвечает в формате JSON.
5. Настроены базовые middleware: `express.json()` и `cors()`.

# Тикет 35: Настройка подключения к MySQL (настройка пула соединений)

## Описание

Интеграция серверного приложения с реляционной базой данных, развернутой в рамках Тикета 25. Для обеспечения высокой пропускной способности и отказоустойчивости при параллельных запросах (согласно спецификации) необходимо использовать механизм пула потоков (Connection Pool).

## Технические требования

- **Драйвер:** Установить пакет `mysql2` (использовать модуль `mysql2/promise` для поддержки современного `async/await` синтаксиса).
- **Конфигурация (`src/config/db.ts`):**
    - Создать пул соединений `mysql.createPool(...)`.
    - Параметры подключения должны браться строго из переменных окружения (`.env`): `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`.
    - Дополнительные настройки пула: `waitForConnections: true`, `connectionLimit: 10` (базовое значение), `queueLimit: 0`.
- **Проверка соединения:**
    - Реализовать функцию проверки связи с БД при старте сервера (например, выполнить простой запрос `SELECT 1`). При успешном соединении выводить сообщение в консоль, при ошибке — логировать причину и, при необходимости, останавливать запуск сервера.

## Критерии приемки (DoD)

1. Пакет `mysql2` установлен.
2. Файл конфигурации БД создан и корректно читает данные из `.env`.
3. Пул соединений успешно инициализируется при запуске приложения.
4. При старте сервера в консоли появляется сообщение об успешном подключении к MySQL (или понятная ошибка, если БД недоступна).

# Тикет 36: Настройка глобального обработчика ошибок REST API

## Описание

Согласно нефункциональным требованиям (п. 1.8.4), система должна корректно обрабатывать исключения (например, при загрузке файлов некорректного формата, возвращая HTTP 400), исключая аварийную остановку (краш) серверного процесса. Для этого необходимо внедрить глобальный Error Handling Middleware в Express.

## Технические требования

- **Пользовательский класс ошибки (`src/utils/AppError.ts`):**
    - Создать класс `AppError`, наследуемый от встроенного `Error`.
    - Класс должен принимать параметры `statusCode` (number) и `message` (string).
- **Middleware (`src/middlewares/errorHandler.ts`):**
    - Написать функцию-обработчик с четырьмя аргументами: `(err, req, res, next)`.
    - Логика: Если ошибка является экземпляром `AppError`, возвращать указанный статус-код и сообщение. Если это непредвиденная системная ошибка, логировать её в консоль и возвращать HTTP 500 (Internal Server Error) со стандартным сообщением ("Что-то пошло не так").
- **Интеграция:**
    - Подключить этот middleware в `app.ts` (или `index.ts`) **в самом конце** цепочки роутов.
- **Обработка асинхронных ошибок:**
    - Настроить перехват `unhandledRejection` и `uncaughtException` на уровне процесса Node.js для логирования непредвиденных крашей.

## Критерии приемки (DoD)

1. Класс кастомной ошибки `AppError` реализован и доступен для импорта.
2. Middleware обработки ошибок подключен в главном файле приложения.
3. Проведен тест: брошен искусственный `new AppError(400, 'Bad Request')` внутри тестового роута; сервер не упал, а клиент получил JSON `{ "status": "error", "message": "Bad Request" }`.
4. Глобальные обработчики `uncaughtException` и `unhandledRejection` настроены.
