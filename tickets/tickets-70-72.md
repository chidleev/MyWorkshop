# Тикет 70: Интеграция складского модуля: связывание интерфейсов прихода ТМЦ и физических остатков с API

## Описание

Необходимо «оживить» рабочее пространство Кладовщика (Фронтенд-тикеты 16, 17, 18). Цифровая панель физических остатков, модальное окно прихода фурнитуры и история транзакций должны перестать использовать mock-данные и начать работать напрямую с базой данных через серверное API (Бэкенд-тикеты 52, 55, 57).

## Технические требования

- **API-сервис (`src/api/inventory.ts`):**
    - Методы: `fetchInventory(params?: { search?: string })`, `fetchTransactions(params)`, `registerIncomingStock(materialId, quantity)`.
- **Интеграция в `InventoryView.vue` (Остатки):**
    - В хуке `onMounted` загружать данные через `fetchInventory()`.
    - Удалить mock-стейт. Реализовать локальный поиск по полученному массиву (или передавать query-параметр поиска на сервер).
- **Интеграция в `IncomingStockModal.vue` (Приход):**
    - Заменить mock-обновление стейта на вызов `registerIncomingStock()`.
    - Во время запроса блокировать кнопку отправки.
    - При успешном ответе (HTTP 200) закрыть модалку, показать Toast «Успех» и обязательно заново вызвать `fetchInventory()` в родительском компоненте для актуализации цифр на экране.
- **Интеграция в `TransactionsView.vue` (Журнал):**
    - Связать таблицу с методом `fetchTransactions()`. Реализовать передачу параметров фильтрации (тип операции, ID заказа) в API.

## Критерии приемки (DoD)

1. Мок-данные полностью удалены из складских компонентов.
2. При оформлении прихода через модальное окно данные уходят на сервер, и таблица остатков реактивно обновляется.
3. Журнал транзакций корректно отображает историю с бэкенда (с учетом джоина имен материалов).
4. Ошибки (например, отрицательное значение прихода) перехватываются глобальным обработчиком и выводятся в UI.

# Тикет 71: Интеграция модуля снабжения: связывание интерфейса дефицита с API отчетов

## Описание

Рабочее пространство Закупщика (Фронтенд-тикет 19) должно получать данные от агрегирующего эндпоинта дефицита (Бэкенд-тикет 56). Это позволит снабженцу видеть реальную картину нехватки материалов, сформированную на основе автоматических списаний при запуске заказов в производство.

## Технические требования

- **API-сервис (`src/api/inventory.ts` или `src/api/buyer.ts`):**
    - Добавить метод `fetchDeficitReport(params?: { include_reserves?: boolean })`, отправляющий `GET /api/inventory/deficit` с query `include_reserves` при необходимости.
- **Интеграция в `DeficitReportView.vue`:**
    - При монтировании — `fetchDeficitReport({ include_reserves })` (переключатель резерва на UI); при необходимости предварительно `isServerAvailable` (см. реализацию экрана).
    - Таблица через `AppDataTable`; пользователь может скорректировать количество «к закупке» per row; итоговый бюджет — сумма строк на клиенте.
    - Кнопка экспорта CSV (клиентская выгрузка `deficit-report.csv`, см. Тикет 19).
- **Обработка пустого состояния (Empty State):**
    - Если бэкенд возвращает пустой массив (дефицита нет), показать позитивное сообщение: «Все материалы в наличии. Дефицита нет!» (например, с иконкой зеленой галочки).

## Критерии приемки (DoD)

1. Экран закупщика делает `GET /api/inventory/deficit` и показывает только позиции, которые сервер отфильтровал как **недостаточный запас** (условие на бэкенде: выбранная метрика **`<= 0`**, не строго `< 0`).
2. Суммы строк и общий бюджет на экране согласованы с `suggested_qty`, `base_cost` и пользовательскими правками количества; при экспорте используются те же числа, что в таблице.
3. Пустое состояние (отсутствие дефицита) корректно и красиво обрабатывается в интерфейсе.
4. Данные защищены: запрос от лица Монтажника вернет 403 и перенаправит юзера.

# Тикет 72: Интеграция модулей руководителя: получение и отрисовка графиков рентабельности и загрузки

## Описание

Завершающий шаг интеграции фронтенда — подключение дашбордов Руководителя (Фронтенд-тикеты 20, 21, 22) к сложным аналитическим SQL-эндпоинтам (Бэкенд-тикеты 58, 60, 61). Интерфейс должен получить агрегированные данные и визуализировать их с помощью графиков и сводных таблиц.

## Технические требования

- **API-сервис (`src/api/director.ts`):**
    - `fetchGlobalOrders`, `fetchDirectorOrderDetails`, `fetchWorkload`, `fetchProfitability` — соответствующие `GET /api/director/...`.
- **Интеграция `GlobalMonitoringView.vue`:** как в Тикете 20; переход на детальную карточку руководителя при необходимости UX.
- **Интеграция `DirectorOrderDetailView.vue`:** загрузка `fetchDirectorOrderDetails(orderId)` и отображение блоков заказа, спецификации, медиа, истории. В блоке медиа: превью-картинка только для типичных расширений изображений; для PDF и прочих файлов — плитка со ссылкой «Открыть / скачать» (чек/акт после завершения заказа).
- **Интеграция `WorkloadDashboardView.vue` (Загрузка цеха):**
    - Получить JSON-объект загрузки от `fetchWorkload()` (после `isServerAvailable` в реализации).
    - Отрисовать нативные прогресс-бары по счётчикам статусов (см. Тикет 21); отдельный компонент `vue-chartjs` **не** обязателен.
- **Интеграция `ProfitabilityReportView.vue` (Рентабельность):**
    - Вызвать `fetchProfitability()`.
    - Связать полученные финансовые метрики (выручка, себестоимость, маржа) с карточками KPI (Key Performance Indicators) вверху экрана и заполнить таблицу закрытых проектов.

## Критерии приемки (DoD)

1. Мок-вычисления полностью удалены из компонентов руководителя.
2. Прогресс-бары загрузки линий корректно отрисовываются на основе реальных счетчиков из базы данных.
3. Финансовый отчет точно отображает маржинальность проектов, полученную с бэкенда.
4. Глобальные фильтры (например, "только просроченные") отправляют корректный запрос к API и перерисовывают канбан-доску.
5. Интеграция не вызывает ошибок рендеринга (использовать `v-if` на время загрузки — без canvas-графиков по умолчанию).
