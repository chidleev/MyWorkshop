# Тикет 46: Реализация «Сервиса калькуляции»: расчет итоговой стоимости и запись спецификации (`specification_items`)

## Описание

После успешного сопоставления артикулов из загруженного CSV-файла со складской базой (Тикет 45), система должна рассчитать продажную стоимость каждой позиции и массово сохранить весь состав изделия в таблицу `specification_items`.

## Технические требования

- **Сервис (`src/services/PricingService.ts`):**
    - На основе подготовленного массива данных реализовать расчет: `sale_price = required_quantity * base_cost` (согласно п. 1.8.3.2 спецификации).
    - Рассчитать общую итоговую сумму заказа (`totalOrderCost`) путем сложения всех вычисленных `sale_price`. Учитывать работу с типом данных DECIMAL (избегать погрешностей чисел с плавающей точкой в JS, использовать `Number.toFixed(2)` или специализированную библиотеку вроде `decimal.js`).
- **Сохранение (SQL):**
    - Перед вставкой нового состава удалить старые строки: `DELETE FROM specification_items WHERE order_id = ?`.
    - Batch Insert полей: `order_id`, `material_id`, `required_quantity`, `sale_price` (значение `sale_price` = `required_quantity * base_cost` с округлением до 2 знаков на строку).
    - _Оптимизация:_ `INSERT INTO specification_items (...) VALUES (?), (?), ...` одним запросом.
- **Субтотал материалов и цена клиента:**
    - `materials_subtotal` = сумма всех `sale_price` по заказу.
    - Итог в карточке заказа: `total_cost` = `computeClientTotalWithMarkup(materials_subtotal, orders.pricing_markup_mode, orders.pricing_markup_value).clientTotal` (режимы и валидация — см. `PricingService`).
- **Транзакционность:**
    - Метод `saveSpecification` принимает `connection` и выполняется в одной транзакции с обновлением `orders.total_cost` (и при необходимости других полей заказа).

## Критерии приемки (DoD)

1. Расчёт строк спецификации и субтотала материалов согласован с полями наценки в `orders`.
2. Реализован Bulk Insert после полной замены строк спецификации для `order_id`.
3. Округления до 2 знаков на денежные поля и до 3 на количества соблюдены на границе JS/SQL.
4. Откат транзакции при ошибке не оставляет «полусохранённой» спецификации.

# Тикет 47: Калькуляция, сохранение спецификации и обновление `total_cost`; предпросмотр и коммит по JSON

## Описание

Импорт спецификации может идти как загрузкой CSV (`multipart`), так и двумя JSON-эндпоинтами для сценария «табличный ввод / правки на клиенте». Во всех случаях после расчёта обновляется `orders.total_cost` с учётом сохранённых в заказе `pricing_markup_mode` и `pricing_markup_value`. Изменение спецификации и наценки разрешено только в статусе заказа `'Новый'`.

## Технические требования

- **`POST /api/orders/:id/specification`** — `multipart/form-data`, поле `file`: потоковый парсинг CSV (Тикеты 43–44), классификация и сохранение (Тикет 46), обновление `total_cost` и `updated_at` в транзакции. Ответ: `SpecificationMutationResponse` (итоги, флаги `committed`, `has_missing_materials`, строки с полями `row_id`, `article`, `name`, `required_quantity`, `unit_price`, `amount`, `missing_material`).
- **`POST /api/orders/:id/specification/preview`** — JSON `{ rows: [{ article, quantity }, ...] }`: тот же конвейер расчёта **без** записи в БД; транзакция не требуется; ответ в том же формате, `committed: false`.
- **`POST /api/orders/:id/specification/commit`** — JSON с тем же массивом `rows`: полная замена `specification_items`, пересчёт `total_cost`, `committed: true`; при любых `missing` материалах — откат и ошибка.
- **`GET /api/orders/:id/specification`** — отдаёт текущую сохранённую спецификацию и агрегаты (`materials_subtotal`, наценка, `markup_amount`, `total_cost`, строки).
- **Транзакции:** `beginTransaction` / `commit` / `rollback` вокруг любых операций, меняющих `specification_items` и сумму заказа.

## Критерии приемки (DoD)

1. Работают маршруты загрузки CSV, preview и commit согласованно с `PricingService` и ограничением по статусу `'Новый'`.
2. `total_cost` в БД всегда соответствует формуле наценки от актуального субтотала материалов.
3. Ошибки парсинга/валидации не оставляют частично обновлённой спецификации.
4. Клиент получает структурированный ответ, достаточный для таблицы в `OrderDetailView` (Тикеты 10, 67).

# Тикет 48: Реализация «Менеджера статусов»: эндпоинт изменения стадии готовности заказа

## Описание

Движение заказа по производственной цепочке фиксируется изменением его статуса. Мастера цеха и менеджеры (через фронтенд) будут использовать этот эндпоинт для перевода заказа, например, из «Новый» в «В производстве» или из «Сборка» в «Готов к отгрузке».

## Технические требования

- **Маршрутизация (`src/routes/statusRoutes.ts` или `orderRoutes.ts`):**
    - Создать роут `PATCH /api/orders/:id/status`.
    - Защитить эндпоинт через `authMiddleware` (доступ: 'Мастер цеха', 'Менеджер', 'Руководитель', 'Монтажник').
- **Контроллер и валидация:**
    - Тело: `{ "status": "<стадия>" }`.
    - Допустимые значения `status` в `StatusManager` (жёсткий `Set`): **`Новый`**, **`В производстве`**, **`Распил`**, **`Кромление`**, **`Присадка`**, **`Сборка`**, **`Готов к отгрузке`**, **`Завершен`**. Иное значение → `400`.
    - Бизнес-правила (фрагменты): нельзя уводить из `'Завершен'`; в `'Завершен'` только из `'Готов к отгрузке'`; без спецификации нельзя в `'В производстве'` и `'Готов к отгрузке'`; в `'Готов к отгрузке'` только если все задачи в `'Завершен'`; возврат в `'Новый'` запрещён, если по заказу уже есть задачи в работе/завершённые; при возврате в `'Новый'` до старта работ удаляются строки `tasks`; и др. — см. реализацию `StatusManager.changeStatus`.
- **Обновление в БД:**
    - Внутри сервиса: `UPDATE orders SET current_stage = ?, updated_at = CURRENT_TIMESTAMP(3) WHERE id = ?` (и запись в `status_histories`, резервы на `'В производстве'` — см. Тикеты 49–50).

## Критерии приемки (DoD)

1. Создан эндпоинт `PATCH /api/orders/:id/status` с ролями из маршрута.
2. Валидация статуса и бизнес-ограничения совпадают с `StatusManager`.
3. При успешном переходе обновляются `current_stage`, `updated_at` и история статусов.
4. Переход в `'Завершен'` после коммита запускает цепочку: `NotificationService.generateDocuments` (PDF через `ClosingDocumentsService` + записи в `media_files`) и `sendCompletionEmail` (асинхронно после транзакции; ошибки логируются).
