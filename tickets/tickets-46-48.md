# Тикет 46: Реализация «Сервиса калькуляции»: расчет итоговой стоимости и запись спецификации (`specification_items`)

## Описание

После успешного сопоставления артикулов из загруженного CSV-файла со складской базой (Тикет 45), система должна рассчитать продажную стоимость каждой позиции и массово сохранить весь состав изделия в таблицу `specification_items`.

## Технические требования

- **Сервис (`src/services/PricingService.ts`):**
    - На основе подготовленного массива данных реализовать расчет: `sale_price = required_quantity * base_cost` (согласно п. 1.8.3.2 спецификации).
    - Рассчитать общую итоговую сумму заказа (`totalOrderCost`) путем сложения всех вычисленных `sale_price`. Учитывать работу с типом данных DECIMAL (избегать погрешностей чисел с плавающей точкой в JS, использовать `Number.toFixed(2)` или специализированную библиотеку вроде `decimal.js`).
- **Сохранение (SQL):**
    - Сформировать запрос для множественной вставки (Batch Insert) в таблицу `specification_items` полей: `order_id`, `material_id`, `required_quantity`, `sale_price`.
    - _Оптимизация:_ Использовать синтаксис `INSERT INTO specification_items (...) VALUES (?), (?), ...` для вставки всех строк за одно обращение к БД, а не в цикле, чтобы уложиться в норматив до 3 секунд.
- **Транзакционность:**
    - Метод сохранения должен принимать объект соединения (`connection`), чтобы выполняться в рамках единой SQL-транзакции (совместно с обновлением суммы заказа из Тикета 47).

## Критерии приемки (DoD)

1. Написан алгоритм расчета финальной цены каждой детали и общей стоимости заказа.
2. Реализован Bulk Insert (массовая вставка) строк спецификации в базу данных.
3. Ошибки JS-округлений (например, `0.1 + 0.2 = 0.30000000000000004`) корректно нивелируются до 2 знаков после запятой перед отправкой в СУБД.
4. Проверен сценарий отката (Rollback), если вставка спецификации завершается ошибкой.

# Тикет 47: Реализация обновления поля `total_cost` в таблице `orders` по итогам калькуляции

## Описание

Финальный шаг процесса импорта CSV. После того как спецификация разобрана, оценена и сохранена в базу данных, необходимо обновить общую стоимость договора в главной карточке заказа и вернуть сформированный результат клиентскому веб-приложению.

## Технические требования

- **Контроллер (`src/controllers/OrderController.ts`):**
    - Связать выполнение Тикетов 43, 44, 45, 46 в единый процесс внутри `POST /api/orders/:id/specification`.
    - Инициализировать SQL-транзакцию (`connection.beginTransaction()`).
    - После успешной вставки позиций в `specification_items` выполнить запрос:
      `UPDATE orders SET total_cost = ? WHERE id = ?`.
    - Сделать `connection.commit()` или `connection.rollback()` в блоке `catch`.
- **Формат ответа (HTTP 200 OK):**
    - Вернуть клиенту JSON с итоговой ценой и массивом обработанных деталей (для отрисовки таблицы на фронтенде из Тикета 10).
    - Структура ответа: `{ status: "success", data: { total_cost: 150000.00, items: [...] } }`.

## Критерии приемки (DoD)

1. Выполнение парсинга, сохранения спецификации и обновления суммы заказа обернуто в единую SQL-транзакцию.
2. Поле `total_cost` в таблице `orders` успешно обновляется.
3. При любой ошибке в процессе парсинга или вставки БД остается в консистентном состоянии (ни одна строка не сохраняется).
4. Клиенту возвращается корректный HTTP-ответ с рассчитанными данными.

# Тикет 48: Реализация «Менеджера статусов»: эндпоинт изменения стадии готовности заказа

## Описание

Движение заказа по производственной цепочке фиксируется изменением его статуса. Мастера цеха и менеджеры (через фронтенд) будут использовать этот эндпоинт для перевода заказа, например, из «Новый» в «В производстве» или из «Сборка» в «Готов к отгрузке».

## Технические требования

- **Маршрутизация (`src/routes/statusRoutes.ts` или `orderRoutes.ts`):**
    - Создать роут `PATCH /api/orders/:id/status`.
    - Защитить эндпоинт через `authMiddleware` (доступ: 'Мастер цеха', 'Менеджер', 'Руководитель', 'Монтажник').
- **Контроллер и Валидация:**
    - Ожидаемое тело запроса: `{ "status": "Новый_статус" }`.
    - Валидировать передаваемый статус по заранее заданному словарю допустимых стадий (enum: "В обработке", "В производстве", "Распил", "Сборка", "Готов к отгрузке", "Завершен"). При несовпадении — возвращать 400 Bad Request.
- **Обновление в БД:**
    - Выполнить SQL: `UPDATE orders SET current_stage = ? WHERE id = ?`.

## Критерии приемки (DoD)

1. Создан эндпоинт `PATCH /api/orders/:id/status`.
2. Эндпоинт защищен авторизацией, роль извлекается из токена.
3. Происходит строгая валидация входящего строкового значения статуса.
4. Поле `current_stage` в таблице `orders` успешно обновляется при вызове метода.
