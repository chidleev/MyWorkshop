# Тикет 49: Реализация логики записи изменений этапа в таблицу `status_histories`

## Описание

Для обеспечения прозрачности процесса и возможности Руководителю/Клиенту отслеживать хронологию работы над заказом (согласно п. 2.7.1.5), любое изменение статуса должно фиксироваться в журнале истории. Эндпоинт из Тикета 48 должен быть расширен для ведения аудита.

## Технические требования

- **Сервис (`src/services/StatusManager.ts`):**
    - Перенести логику смены статуса (`PATCH /api/orders/:id/status`) в выделенный сервис-менеджер.
    - Все операции по смене статуса должны выполняться строго внутри SQL-транзакции.
- **Логика аудита:**
    - После обновления поля `current_stage` в таблице `orders`, выполнить запрос: `INSERT INTO status_histories (order_id, stage_name, employee_ext_id) VALUES (?, ?, ?)`.
    - Значение `employee_ext_id` брать из `req.user.id` (ID сотрудника, полученный из токена CRM).
- **Интеграция с контроллером:**
    - Контроллер вызывает метод `StatusManager.changeStatus(orderId, newStatus, userId, connection)`.

## Критерии приемки (DoD)

1. При любом успешном изменении статуса заказа создается соответствующая запись в `status_histories`.
2. Обновление заказа и запись в историю объединены в одну транзакцию (одно без другого не сохраняется).
3. Автор изменения (ID сотрудника) корректно фиксируется в БД.
4. Отсутствуют дублирующиеся или пустые записи при попытке сменить статус на тот же самый (добавить проверку `if (oldStatus === newStatus) return;`).

# Тикет 50: Реализация генерации задач в таблицу `tasks` при смене статуса

## Описание

Внедрение автоматизации планирования для производственного цеха. Когда заказ переводится на этап «В производстве», система не должна ждать ручной раздачи поручений. Она должна автоматически сгенерировать набор базовых сменных заданий (распил, кромление, присадка, сборка), чтобы Мастер цеха сразу увидел их на своем табло (Фронтенд-тикет 11).

## Технические требования

- **Сервис (`src/services/StatusManager.ts`):**
    - Добавить слушатель (или прямое условие в код) на событие: `if (newStatus === 'В производстве')`.
- **Генерация задач (SQL):**
    - В рамках текущей транзакции выполнить `INSERT` в таблицу `tasks` нескольких стандартных операций для переданного `order_id`.
    - Набор создаваемых операций по умолчанию: "Распил", "Кромление", "Присадка", "Сборка".
    - Начальное состояние для каждой задачи: `progress_status = 'Ожидает'`.
    - Поле `master_ext_id` оставить `NULL` (мастер будет назначать себя сам, когда возьмет задачу в работу).
- **Защита от дубликатов:**
    - Перед генерацией задач убедиться, что для данного `order_id` задачи еще не были созданы (на случай случайного возврата заказа в предыдущий статус и повторного перевода «В производство»).

## Критерии приемки (DoD)

1. При переводе заказа в статус «В производстве» в таблице `tasks` автоматически появляются 4 базовые технологические операции.
2. Новые задачи создаются со статусом «Ожидает».
3. Генерация задач выполняется внутри общей транзакции изменения статуса.
4. Повторный перевод заказа в этот статус не приводит к созданию дубликатов задач.

# Тикет 51: Реализация эндпоинта загрузки файлов (фотоотчетов) и сохранения ссылок в `media_files`

## Описание

Обеспечение функционала для Монтажников по завершению сделки (Сценарий: п. 2.3.7 спецификации). После сборки мебели на объекте монтажник прикрепляет фотоотчет. Сервер должен принять файл, сохранить его (на диск или в мок-хранилище) и записать защищенную ссылку на него в базу данных.

## Технические требования

- **Маршрутизация (`src/routes/mediaRoutes.ts` или `orderRoutes.ts`):**
    - Создать роут `POST /api/orders/:id/media`.
    - Защитить через `authMiddleware` (разрешить роли 'Монтажник', 'Менеджер', 'Руководитель').
- **Обработка файлов (Multer):**
    - Настроить `multer` для сохранения файлов на диск (создать папку `uploads/` в корне проекта).
    - Ограничить принимаемые форматы: только изображения (`image/jpeg`, `image/png`).
    - Генерировать уникальные имена файлов (например, через `uuid` или добавление `Date.now()`).
- **Логика контроллера:**
    - Ожидать `multipart/form-data` с полем `file` и текстовым полем `file_type` (по умолчанию: "Фото").
    - Сформировать относительный или абсолютный URL-путь (например, `/uploads/filename.jpg`).
    - Выполнить SQL: `INSERT INTO media_files (order_id, file_type, secure_link) VALUES (?, ?, ?)`.
- **Раздача статики (Express):**
    - Настроить в `app.ts` отдачу статических файлов из директории `uploads`: `app.use('/uploads', express.static('uploads'))`.

## Критерии приемки (DoD)

1. Эндпоинт загрузки медиафайлов создан и корректно принимает изображения.
2. Файлы успешно сохраняются в локальную директорию `uploads/`.
3. В таблице `media_files` создается запись со ссылкой на сохраненный файл, привязанная к `order_id`.
4. Загруженный файл можно открыть в браузере по прямой ссылке, генерируемой сервером (через `express.static`).
5. При попытке загрузить не изображение (например, `.pdf` или `.exe`) сервер возвращает 400 Bad Request.
