# Тикет 49: Реализация логики записи изменений этапа в таблицу `status_histories` и синтетические записи по задачам цеха

## Описание

Любое изменение статуса заказа через `StatusManager` фиксируется в `status_histories`. Дополнительно при работе мастера с задачами пишутся «синтетические» подписи вида `Распил (Ожидание)` / `Кромление (В работе)` через `StatusManager.syntheticTaskHistoryLabel`, чтобы журнал отражал прогресс операций.

## Технические требования

- **Сервис (`src/services/StatusManager.ts`):**
    - Все операции смены статуса заказа — в SQL-транзакции.
    - После обновления `orders.current_stage`: `INSERT INTO status_histories (order_id, stage_name, employee_ext_id) VALUES (?, ?, ?)` с `employee_ext_id = req.user.id` из CRM-токена.
    - При переходе в `'В производстве'`: если ещё нет строк в `tasks` для `order_id`, вставить четыре операции (`Распил`, `Кромление`, `Присадка`, `Сборка`) со статусом `'Ожидает'`; затем вызвать `InventoryService.syncReserveForOrder` (резерв материалов по спецификации — см. Тикет 50).
    - При первом переходе в производство добавить в историю подпись по первой задаче (ожидающей/в работе) — см. код после генерации задач.
    - Если новый статус совпадает со старым — выйти без записи в историю (ранний `return` после `rollback` пустой транзакции).

## Критерии приемки (DoD)

1. Каждый успешный `PATCH /api/orders/:id/status` даёт ровно одну новую запись истории для смены стадии заказа (плюс дополнительные записи по правилам задач/готовности — как в коде).
2. История и `orders` остаются согласованными при откате транзакции.
3. `employee_ext_id` заполняется из токена.
4. Нет дублирования записей при повторной отправке того же статуса.

# Тикет 50: Резервирование материалов при «В производстве» и фактическое списание при старте первой задачи

## Описание

При переводе заказа в `'В производстве'` система **не** уменьшает колонку остатка в `materials`. Вместо этого создаётся/обновляется набор строк `inventory_transactions` с типом **`Резерв`** (отрицательный `quantity_change` на пару `order_id` + `material_id`, без дублей при повторном синхроне). Фактическое уменьшение физического остатка происходит, когда первая (или очередная) задача цеха переходит в **`В работе`**: строки резерва того же заказа обновляются до типа **`Списание`** с актуальным `tx_date`, при этом перед переходом вызывается `assertReserveConversionWontCauseNegativePhysicalStock` — если после конверсии физический остаток стал бы отрицательным, возвращается **409** и транзакция откатывается.

## Технические требования

- **`InventoryService.syncReserveForOrder`:** агрегировать потребность по `specification_items`, синхронизировать одну строку резерва на материал; удалять резервы по материалам, выпавшим из спецификации; не трогать уже существующие **`Списание`** по заказу.
- **`InventoryService.convertReserveToConsumptionForOrder`:** для всех резервов заказа выполнить `UPDATE inventory_transactions SET tx_type = 'Списание', tx_date = CURRENT_TIMESTAMP(3) WHERE ...`.
- **`InventoryService.assertReserveConversionWontCauseNegativePhysicalStock`:** для каждого материала с резервом заказа вычислить текущий физический остаток по формуле «снимок + списания после `stock_snapshot_at`» и убедиться, что `physical + sum(reserve quantity_change) >= 0`.
- **Цех (`WorkshopController.patchTaskStatus`):** при `nextStatus === 'В работе'` вызвать проверку, затем после `UPDATE tasks` — конверсию резерва в списание; параллельно обновлять `orders.current_stage` по правилам «активная операция / очередь / все задачи завершены → Готов к отгрузке» и писать историю.

## Критерии приемки (DoD)

1. При `'В производстве'` создаются задачи (если их ещё не было) и синхронизируется резерв по спецификации.
2. При взятии задачи в работу негативный физический остаток блокируется с понятной ошибкой.
3. После успешного старта задачи движения в журнале имеют тип `Списание` с корректным временем.
4. Повторный перевод в производство не плодит дубли задач и согласует резерв с актуальной спецификацией.

# Тикет 51: Реализация эндпоинта загрузки файлов (фотоотчетов) и сохранения ссылок в `media_files`

## Описание

Обеспечение функционала для Монтажников по завершению сделки (Сценарий: п. 2.3.7 спецификации). После сборки мебели на объекте монтажник прикрепляет фотоотчет. Сервер должен принять файл, сохранить его (на диск или в мок-хранилище) и записать защищенную ссылку на него в базу данных.

## Технические требования

- **Маршрутизация (`src/routes/mediaRoutes.ts` или `orderRoutes.ts`):**
    - Создать роут `POST /api/orders/:id/media`.
    - Защитить через `authMiddleware` (разрешить роли 'Монтажник', 'Менеджер', 'Руководитель').
- **Обработка файлов (Multer):**
    - Использовать `multer.diskStorage` с **абсолютным** каталогом из `ensureRuntimeDirectories()` / `uploadsPath` в `src/config/paths.ts` (корень процесса + `uploads/`, каталог создаётся при старте приложения).
    - Ограничить принимаемые форматы: только изображения (`image/jpeg`, `image/png`).
    - Имена файлов: уникальный префикс (`Date.now()` + санитизированное оригинальное имя).
- **Логика контроллера:**
    - Ожидать `multipart/form-data` с полем `file` и текстовым полем `file_type` (по умолчанию: "Фото").
    - В `secure_link` сохранять относительный путь вида `uploads/<filename>` (или с ведущим `/` — единообразно с раздачей статики и `resolveManagedFilePath` при удалении).
    - Выполнить SQL: `INSERT INTO media_files (order_id, file_type, secure_link) VALUES (?, ?, ?)`.
- **Раздача статики (Express) в `app.ts`:**
    - `app.use('/uploads', express.static(uploadsPath))`.
    - Дополнительно: `app.use('/documents', express.static(documentsPath))` для отдачи **реальных PDF** закрывающих документов из каталога `documents/` (генерация — `ClosingDocumentsService`, оркестрация и `media_files` — `NotificationService.generateDocuments`; пути — `src/config/paths.ts`).
    - Для загрузки картинок с другого origin при CORS: в `helmet` задать `crossOriginResourcePolicy: { policy: 'cross-origin' }`.

## Критерии приемки (DoD)

1. Эндпоинт загрузки медиафайлов создан и корректно принимает изображения.
2. Файлы успешно сохраняются в локальную директорию `uploads/`.
3. В таблице `media_files` создается запись со ссылкой на сохраненный файл, привязанная к `order_id`.
4. Загруженный файл доступен по URL с префиксом `/uploads/...` и корректно резолвится на фронтенде через `resolveBackendAssetUrl` при полном `API_BASE_URL`.
5. При попытке загрузить не изображение (например, `.pdf` или `.exe`) сервер возвращает 400 Bad Request.
