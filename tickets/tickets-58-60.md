# Тикет 58: Реализация эндпоинта получения глобального реестра сделок для руководителя

## Описание

Этот эндпоинт обеспечивает работу главного дашборда Руководителя (Фронтенд-тикет 20). В отличие от стандартного получения списка заказов менеджером (Тикет 40), данный запрос должен извлекать информацию по всей мастерской без привязки к конкретному сотруднику, чтобы обеспечить глобальный контроль над производственными сроками.

## Технические требования

- **Маршрутизация (`src/routes/directorRoutes.ts`):**
    - Подключить `router.use(authMiddleware, requireRole(['Руководитель']))` для всех методов контроллера.
    - `GET /api/director/orders` — глобальный список: `JOIN clients`, в выборку входят все поля `orders` (`o.*`) плюс `full_name`, `phone` клиента.
    - `GET /api/director/orders/:id` — карточка заказа для вложенного экрана руководителя: объект `data` с полями `order` (все колонки заказа + контакт клиента), массивы `specification` (позиции с `article`, `name`, `required_quantity`, `sale_price`), `media`, `status_histories`.
- **Фильтры query для списка:** `manager_id`, `status` (равенство `current_stage`), `is_overdue=true` → условие `o.target_date < CURDATE() AND o.current_stage != 'Завершен'` (корректно для типа `DATETIME` цели сдачи).
- **Сортировка:** `ORDER BY o.target_date ASC`.

## Критерии приемки (DoD)

1. Работают `GET /api/director/orders` и `GET /api/director/orders/:id` только для роли `Руководитель`.
2. Фильтры `manager_id`, `status`, `is_overdue` формируют ожидаемый `WHERE`.
3. Детальный ответ содержит спецификацию, медиа и историю статусов.
4. Список отсортирован по `target_date`.

# Тикет 59: Реализация эндпоинтов получения сменных заданий (для мастера) и нарядов на установку (для монтажника)

## Описание

Данный тикет обеспечивает данными рабочие планшеты сотрудников цеха (Фронтенд-тикеты 11, 12) и выездных бригад (Фронтенд-тикеты 13, 14). Необходимо создать два эндпоинта, которые будут возвращать списки задач и объектов на текущий день или ближайшую смену.

## Технические требования

- **Дублирование префикса в `app.ts`:** `app.use('/api/workshop', workshopRoutes)` и `app.use('/api/installer', workshopRoutes)` — один и тот же роутер; фронтенд в `src/api/installer.ts` и `src/api/workshop.ts` использует пути с **`/api/workshop/...`**.
- **Эндпоинт для Мастера цеха (`GET /api/workshop/tasks`):**
    - Роли: `Мастер цеха`, `Руководитель`.
    - `JOIN orders`, `JOIN clients`; поля задачи + `agreement_number`, `target_date`, `full_name`.
    - Query `?status=` — фильтр по `tasks.progress_status` (например `Ожидает`, `В работе`).
    - Для роли `Мастер цеха`: по `req.user.id` выбирается допустимая **одна** операция из мапы `MASTER_OPERATION_BY_ID` (например `master-cutting-001` → только `Распил`); если id не известен — возвращается пустой список.
    - Показывать только задачи, для которых **все предыдущие** по порядку операций (`Распил` → `Кромление` → `Присадка` → `Сборка`) уже в статусе `Завершен` (условие `NOT EXISTS` по «предыдущим» задачам с `progress_status <> 'Завершен'`).
    - Сортировка: `ORDER BY (progress_status = 'Ожидает') DESC, o.target_date ASC`.
- **Смена статуса задачи:** `PATCH /api/workshop/tasks/:id/status` с телом `{ "status": "Ожидает" | "В работе" | "Завершен" }` — **не** путать с `PATCH /api/orders/:id/status`. Внутри — транзакция: проверка переходов, блокировка если предыдущий этап не завершён, при переходе в `В работе` — проверка остатков и конверсия резерва в списание (Тикет 50), обновление `orders.current_stage` и записи в `status_histories` по правилам `WorkshopController`.
- **Эндпоинт для монтажника и др. (`GET /api/workshop/deployments`):**
    - Роли: `Монтажник`, `Руководитель`, `Менеджер`.
    - Заказы в стадиях **`Готов к отгрузке`** или **`Сборка`**; `JOIN clients`; `LEFT JOIN media_files` (в выборке используется одна связанная ссылка `secure_link` — при нескольких файлах возможны дубли строк, фронт дедуплицирует при необходимости).
    - Сортировка: `ORDER BY o.target_date ASC`.

## Критерии приемки (DoD)

1. Реализованы `GET /api/workshop/tasks`, `PATCH /api/workshop/tasks/:id/status`, `GET /api/workshop/deployments` с указанными ролями и бизнес-правилами очереди операций.
2. Эндпоинт `tasks` корректно отдает операции с привязкой к заказам.
3. Эндпоинт `deployments` формирует массив «маршрутных листов» с адресами и ссылками на рендеры.
4. SQL-запросы оптимизированы (используют индексы по статусам).

# Тикет 60: Реализация эндпоинта калькуляции рентабельности по закрытому проекту

## Описание

Самый важный финансовый аналитический эндпоинт для Руководителя (Фронтенд-тикет 22). Он должен сопоставлять доходы (сумму, которую заплатил клиент) с расходами (базовой стоимостью списанных материалов), вычисляя чистую валовую прибыль и маржинальность каждого завершенного заказа.

## Технические требования

- **Маршрутизация (`src/routes/directorRoutes.ts`):**
    - Создать роут `GET /api/director/profitability`.
    - Защитить через `authMiddleware` (строго только 'Руководитель').
- **Логика расчета (Сложный SQL-запрос):**
    - Написать агрегирующий запрос. Базовая таблица: `orders` (где `current_stage = 'Завершен'`).
    - Сделать `JOIN specification_items` (чтобы получить состав) и `JOIN materials` (чтобы получить `base_cost`).
    - **Группировка:** Сгруппировать по `orders.id`.
    - **Формулы в SELECT:**
        - Выручка: `orders.total_cost` AS `revenue`.
        - Себестоимость: `SUM(specification_items.required_quantity * materials.base_cost)` AS `cogs` (Cost of Goods Sold).
        - Валовая прибыль: `orders.total_cost - SUM(specification_items.required_quantity * materials.base_cost)` AS `profit`.
- **Формат ответа:**
    - Вернуть массив объектов: `[ { order_id, agreement_number, client_name, revenue, cogs, profit, margin_percent }, ... ]`.
    - Процент маржинальности (`margin_percent`) можно рассчитать прямо в SQL или на уровне Node.js сервиса: `(profit / revenue) * 100`.

## Критерии приемки (DoD)

1. Эндпоинт `GET /api/director/profitability` создан и надежно защищен.
2. Агрегирующий SQL-запрос выполняется корректно и без математических ошибок (учитывая работу с типом DECIMAL).
3. В ответ попадают только заказы со статусом «Завершен».
4. Выручка, себестоимость, чистая прибыль и процент рентабельности рассчитываются абсолютно точно.
