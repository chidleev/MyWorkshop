# Тикет 9: Создание компонента загрузки CSV-спецификации (с поддержкой Multipart/form-data)

## Описание

Согласно спецификации, Менеджер должен иметь возможность загружать файл спецификации (формат CSV, до 2000 строк) напрямую в систему. Это избавляет от ручного переноса данных из конструкторских программ. Компонент должен обеспечивать удобный интерфейс загрузки (Drag & Drop или стандартный input) и обрабатывать состояния (загрузка, успех, ошибка).

## Технические требования

- **Компонент (`src/components/Orders/SpecificationUploader.vue`):**
    - Реализовать зону для перетаскивания файла (drag-and-drop) и кнопку «Выбрать файл».
    - Ограничить выбор файлов только форматом `.csv`.
- **Логика работы:**
    - При выборе файла формировать объект `FormData` и добавлять в него файл (подготовка к отправке с `Content-Type: multipart/form-data`).
    - Реализовать mock-запрос (через `setTimeout`) для имитации задержки парсинга на сервере.
    - Обработка состояний:
        - Пока идет запрос — показывать индикатор загрузки (spinner/skeleton).
        - При неверном формате — отображать ошибку "HTTP 400 Bad Request" или понятное пользователю сообщение ("Некорректный формат файла").
- **Связь с родителем:**
    - После "успешной" загрузки компонент должен эмитить событие `upload-success` и передавать mock-массив разобранных позиций и рассчитанную итоговую цену (для использования в Тикете 10).

## Критерии приемки (DoD)

1. Компонент загрузки сверстан и корректно реагирует на выбор/перетаскивание файла.
2. Реализована клиентская валидация: загрузить можно только `.csv`.
3. Присутствует визуальный отклик (загрузка, успех, ошибка).
4. `FormData` формируется правильно (можно проверить через `console.log` содержимого).

# Тикет 10: Реализация экрана отображения извлеченного состава мебели и итоговой стоимости заказа

## Описание

Сразу после импорта спецификации из CSV (Тикет 9) система автоматически рассчитывает стоимость. Менеджеру необходимо увидеть этот результат: подробный список комплектующих, их количество, цену и общую итоговую сумму заказа, чтобы быстро озвучить её клиенту.

## Технические требования

- **Компонент/Страница (`src/views/Manager/OrderDetailView.vue`):**
    - Создать страницу детализации конкретного заказа (переход по клику на карточку из канбан-доски).
    - Встроить компонент `SpecificationUploader.vue` (из Тикета 9) на эту страницу.
- **Отображение данных (Спецификация):**
    - Сверстать таблицу (Data Table) `specification_items`, содержащую колонки: «Артикул», «Наименование», «Требуемое кол-во», «Цена за ед.», «Сумма».
    - Реализовать отдельный визуальный блок-акцент «Итоговая стоимость заказа» (total_cost).
- **Логика:**
    - Таблица и итоговая сумма изначально пусты (если спецификация не загружена).
    - При срабатывании события `upload-success` от загрузчика таблица должна реактивно заполняться данными, а общая стоимость пересчитываться и отображаться на экране.

## Критерии приемки (DoD)

1. Создан роут детализации заказа (например, `/orders/:id`).
2. На странице отображаются базовые данные клиента и статус.
3. Таблица спецификации корректно рендерит переданный массив деталей.
4. Итоговая сумма заказа (с точностью до двух знаков после запятой, как DECIMAL) хорошо заметна и обновляется при загрузке данных.

# Тикет 11: Разработка экрана «Табло сменных заданий» с перечнем активных нарядов (Мастер цеха)

## Описание

Необходимо разработать рабочий экран для роли «Мастер цеха». Рабочие на производстве взаимодействуют с системой через планшеты или терминалы. Интерфейс должен быть максимально простым, крупным и наглядным, заменяя собой запутанные бумажные чертежи и маршрутные листы.

## Технические требования

- **Страница (`src/views/Workshop/ShiftTasksView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Мастер цеха'`.
    - Сверстать «Табло» — сетку карточек сменных заданий (`tasks`), назначенных на текущую дату.
- **Содержимое карточки задания:**
    - Номер заказа (связь с клиентским проектом).
    - Название операции (например: "Распил ДСП", "Кромление", "Присадка", "Сборка").
    - Статус прогресса (progress_status).
- **Стейт / Mock-данные:**
    - Создать мок-массив нарядов (маппинг таблицы `tasks` из БД).
    - Реализовать вкладки или фильтры: «Новые», «В работе», «Завершенные».
- **UX/UI нюансы:**
    - Учитывать использование на сенсорных экранах: шрифты должны быть крупными, расстояния между элементами — комфортными для нажатия пальцем.

## Критерии приемки (DoD)

1. Создан защищенный роут `/workshop/tasks`. При логине под Мастером цеха происходит редирект на этот экран.
2. Список сменных заданий рендерится из mock-состояния.
3. Доступна фильтрация задач по статусам.
4. UI адаптирован для просмотра на планшетных устройствах (touch-friendly интерфейс).
