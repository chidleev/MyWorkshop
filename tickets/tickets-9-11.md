# Тикет 9: Компонент загрузки CSV-спецификации

## Описание

Загрузка файла `.csv` на сервер через `FormData` и `POST /api/orders/:id/specification` (см. Тикеты 43–44, 47). Альтернативно родитель может передать уже распарсенные строки в preview/commit (Тикет 67) — компонент может поддерживать оба сценария по UX-потребности.

## Технические требования

- **Компонент (`src/components/Orders/SpecificationUploader.vue`):**
    - Drag-and-drop и выбор файла; только `.csv`.
    - Вызов `uploadSpecification(orderId, file)` из `src/api/orders.ts` с корректным `multipart/form-data`.
    - Состояния: loading, success, error; блокировка двойной отправки.
    - Ошибки 400 — через глобальные уведомления (Тикет 24).
- **События:** по успеху обновить родителя (emit или колбэк) данными ответа `SpecificationMutationResponse`.

## Критерии приемки (DoD)

1. Файл уходит на реальный бэкенд, без `setTimeout`-mock.
2. Клиентская валидация расширения файла.
3. Обработаны сетевые и серверные ошибки.

# Тикет 10: Экран состава заказа, наценка и итоговая цена

## Описание

Страница **`OrderDetailView`** (`/orders/:id`): данные заказа, клиент, стадия; таблица спецификации; блок **`materials_subtotal`**, параметры **`pricing_markup_mode`** / **`pricing_markup_value`**, **`markup_amount`**, итог **`total_cost`**; управление наценкой через **`patchOrderPricingMarkup`** (только стадия **`Новый`** на сервере).

## Технические требования

- Карточка заказа (клиент, договор, стадия, даты): **`useManagerOrdersStore`**, `findById(route.params.id)`; при монтировании страницы вызывать **`ordersStore.ensureLoaded()`** (или `ensureLoaded(true)` после мутаций), чтобы список заказов был загружен с API даже при прямом заходе по URL, а не только после перехода с канбана.
- Таблица колонок из API: артикул, наименование, количество, цена за ед., сумма; строки с `missing_material` визуально отличать.
- **Редактирование состава:** при табличном сценарии — `previewOrderSpecification` / `commitOrderSpecification` с массивом `{ article, quantity }`.
- Отображение дат в интерфейсе с учётом `DATETIME` (`utils/datetime` при наличии).

## Критерии приемки (DoD)

1. Роут `/orders/:id` защищён ролью `Менеджер`.
2. Итоги и наценка соответствуют полям API и пересчитываются после PATCH наценки.
3. Спецификация недоступна для изменения вне стадии `Новый` (сообщения об ошибках с сервера).

# Тикет 11: Табло сменных заданий (Мастер цеха)

## Описание

Страница **`ShiftTasksView`** (`/workshop/tasks`): данные с **`GET /api/workshop/tasks`**, смена статуса задачи через **`PATCH /api/workshop/tasks/:id/status`** (обёртка `updateWorkshopTaskStatus` в `src/api/workshop.ts` с маппингом `new` / `in_progress` / `done` → русские статусы).

## Технические требования

- Карточка задания (`ShiftTaskCard.vue`): отображение операции, договора, клиента, срока; кнопки вызывают API, после успеха — повторная загрузка списка.
- Фильтры по вкладкам статуса прогресса — на клиенте или через query `status`.
- Крупные тач-таргеты.

## Критерии приемки (DoD)

1. Нет mock-массива задач: только ответ API.
2. Роут доступен роли `Мастер цеха`.
3. После действия список консистентен с сервером.
