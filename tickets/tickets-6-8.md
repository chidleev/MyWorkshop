# Тикет 6: Канбан-доска / таблица «Мои заказы» (Менеджер)

## Описание

Главный экран менеджера — реестр сделок с колонками по производственным стадиям (в т.ч. **`Новый`** для черновиков и веб-заявок). Данные загружаются с бэкенда; отображаются договор, клиент, **`target_date`** (дата и время), **`total_cost`**, **`updated_at`**, индикаторы спецификации и прогресса задач из DTO.

## Технические требования

- **Компонент (`src/views/Manager/OrdersView.vue`):**
    - Роут `/orders`, роль `Менеджер` (`meta.allowedRoles` в `router/index.ts`).
    - **Мобильный вид:** колонки-стадии с карточками заказов (как канбан-лента).
    - **Desktop (`md+`):** плотная таблица на **`AppDataTable`** (`hidden md:block`) с сортировкой, слотами ячеек (договор, клиент, дата/время, сумма, меню действий); те же стадии, что и в мобильном канбане.
    - Колонки должны покрывать актуальные стадии из `StatusManager` и **`Новый`**. _Примечание:_ в БД могут остаться строки со стадией **`В обработке`** (старые сиды); на фронтенде в `managerOrders` такая стадия мапится в ту же колонку, что и «новые» заказы (`mapStageToStatus` → `new`), пока запись не переведут через API на допустимый статус.
    - Карточка: `agreement_number`, `full_name`, `target_date`, `total_cost`, при наличии — `production_stage`, счётчики `tasks_*`, флаг спецификации.
    - В шапке раздела — переход на **`/orders/applications`** (веб-заявки), вызов `fetchWebApplications` по необходимости UX.
- **Данные:** `onMounted` → `fetchOrders()` из `src/api/orders.ts`; обработка загрузки и ошибок (axios interceptor / toast).
- **Стилизация:** выделение просрочки на клиенте (сравнение `target_date` с текущим моментом и статусом ≠ `Завершен`), если не используется серверный фильтр.

## Критерии приемки (DoD)

1. Доска строится из ответа `GET /api/orders`, без локального mock-массива заказов.
2. Роут защищён ролью `Менеджер`.
3. Отображаются поля DTO, нужные для операционной работы (в т.ч. время обновления).
4. Адаптивная вёрстка сохранена.

# Тикет 7: Создание новой карточки заказа (модальное окно)

## Описание

Создание сделки через **`POST /api/orders`**: клиент, контакты, опционально договор и срок, стартовая стадия на сервере — **`Новый`**.

## Технические требования

- **Компонент (`src/components/Orders/OrderFormModal.vue`):**
    - Поля: `full_name` (обяз.), `phone`, **`email` (обяз., валидный email — требование API)**, `address`, `agreement_number` (опц.; если пусто — сервер выдаст авто-номер), **`target_date`** — желательно `datetime-local` / ISO-строка совместимая с `DATETIME` на сервере.
    - Для роли **`Руководитель`**: выбор менеджера (`manager_ext_id`) при создании от его имени — по текущей реализации формы.
- **Сабмит:** `createOrder` из `src/api/orders.ts`; по успеху — закрыть модалку, обновить список заказов (`fetchOrders`), toast об успехе.
- **Валидация:** клиентская, согласованная с zod-схемой бэкенда.

## Критерии приемки (DoD)

1. Новый заказ появляется в списке со стадией `Новый` после успешного ответа API.
2. Ошибки 400 от сервера показываются пользователю.
3. Модалка очищается после успеха.

# Тикет 8: Редактирование и удаление заказа

## Описание

Изменение контактов/сроков/договора через **`PUT /api/orders/:id`**, удаление через **`DELETE /api/orders/:id`**. Права владения проверяются на сервере (чужой заказ для менеджера — 403).

## Технические требования

- **UI:** действия из канбана, с **`OrderDetailView`** и из **сайдбара `AppLayout`** на роуте `order-detail` (блок «Управление заказом»: смена стадии, редактирование, отмена); подтверждение перед удалением.
- **API:** `updateOrder`, `deleteOrder` из `src/api/orders.ts`.
- **После успеха:** обновить стор/список или убрать карточку при удалении.

## Критерии приемки (DoD)

1. PUT и DELETE вызывают реальные эндпоинты и обрабатывают 403/404.
2. UI согласован с ответами сервера.
