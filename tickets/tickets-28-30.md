# Тикет 28: Настройка таблиц: создание таблицы `materials` (Складская номенклатура)

## Описание

Создание единого справочника фурнитуры и плитных материалов (согласно п. 2.7.1.3 спецификации). Эта таблица является основой для складского учета, выявления дефицита и расчета базовой себестоимости каждого изделия.

## Технические требования

- **Таблица `materials`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Уникальный системный номер номенклатуры.
    - `article` — `VARCHAR(100)`, `NOT NULL`, `UNIQUE`. Производственный артикул детали (уникальный индекс для быстрого поиска при парсинге CSV).
    - `name` — `VARCHAR(255)`, `NOT NULL`. Текстовое наименование комплектующего.
    - `base_cost` — `DECIMAL(10,2)`, `NOT NULL`. Закупочная стоимость единицы товара (2 знака после запятой для финансов).
    - `current_stock` — `DECIMAL(10,3)`, `NOT NULL`, По умолчанию `0.000`. Фактический физический остаток на полках. _Важно: 3 знака после запятой используются для точного учета метража (погонные метры) или кубатуры/площади листовых материалов._

## Критерии приемки (DoD)

1. Написан и выполнен SQL-скрипт (или миграция) для создания таблицы `materials`.
2. Поля `base_cost` и `current_stock` имеют строгий формат `DECIMAL` с заявленной в ТЗ точностью.
3. Ограничения `NOT NULL` и дефолтное значение `0.000` для остатков отрабатывают корректно.

# Тикет 29: Настройка таблиц: создание таблицы `specification_items` (Позиции спецификации)

## Описание

Создание таблицы для хранения состава будущего изделия (п. 2.7.1.4 спецификации). Каждая запись связывает конкретный клиентский заказ со складским каталогом. Сюда будут сохраняться данные, извлеченные из CSV-файла, для точного расчета продажной цены мебели.

## Технические требования

- **Таблица `specification_items`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Системный номер строки состава.
    - `order_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `orders(id)`. При удалении заказа состав должен удаляться каскадно (`ON DELETE CASCADE`).
    - `material_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `materials(id)` (`ON DELETE RESTRICT`).
    - `required_quantity` — `DECIMAL(10,3)`, `NOT NULL`. Требуемый объем необходимого сырья (3 знака после запятой).
    - `sale_price` — `DECIMAL(10,2)`, `NOT NULL`. Финальная цена продажи конкретной детали клиенту.
- **Индексы и связи:**
    - Создать индексы по `order_id` и `material_id` для быстрого `JOIN` при генерации отчетов.

## Критерии приемки (DoD)

1. Написан и выполнен SQL-скрипт (или миграция) для создания таблицы `specification_items`.
2. Внешние ключи (Foreign Keys) на таблицы `orders` и `materials` настроены и протестированы.
3. Поля с финансовыми и количественными показателями используют формат `DECIMAL`.
4. Каскадное удаление строк спецификации при удалении заказа функционирует верно.

# Тикет 30: Настройка таблиц: создание таблицы `status_histories` (История жизненного цикла)

## Описание

Внедрение таблицы аудита статусов (согласно п. 2.7.1.5 спецификации). Таблица предназначена для регистрации фактов перевода работы на следующие этапы готовности. Это позволит Руководителю и Клиенту видеть хронологию производства изделия.

## Технические требования

- **Таблица `status_histories`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Порядковый номер изменения.
    - `order_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `orders(id)` (`ON DELETE CASCADE`).
    - `stage_name` — `VARCHAR(100)`, `NOT NULL`. Название активированной стадии (например, "В обработке", "Распил", "Сборка").
    - `transition_date` — `DATETIME`, `NOT NULL`, По умолчанию текущее время (`DEFAULT CURRENT_TIMESTAMP`). Точный момент переключения статуса.
    - `employee_ext_id` — `VARCHAR(255)`, `NOT NULL`. Внешний идентификатор профиля сотрудника из корпоративной CRM, который инициировал изменение.
- **Индексы:**
    - Добавить индекс по `order_id` для быстрой загрузки истории конкретной сделки.

## Критерии приемки (DoD)

1. Скрипт создания таблицы `status_histories` успешно выполнен.
2. Внешний ключ на таблицу `orders` установлен корректно.
3. Поле `transition_date` автоматически заполняется датой и временем СУБД в момент создания записи (выполнен INSERT-тест).
4. Таблица присутствует в БД `my_workshop`.
