# Тикет 31: Настройка таблиц: создание таблицы `tasks` (Сменные задания)

## Описание

Создание таблицы для хранения конкретных технологических шагов (операций) в рамках производственного цикла заказа (п. 2.7.1.6 спецификации). Эта таблица будет использоваться Мастером цеха для получения сменных заданий (распил, кромление, присадка, сборка) и фиксации их готовности.

## Технические требования

- **Таблица `tasks`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Уникальный номер операции.
    - `order_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `orders(id)` (`ON DELETE CASCADE`).
    - `operation_name` — `VARCHAR(100)`, `NOT NULL`. Название действия (например, "Распил", "Сборка").
    - `progress_status` — `VARCHAR(100)`, `NOT NULL`. Индикатор завершенности задачи (например, "Ожидает", "В работе", "Завершено").
    - `master_ext_id` — `VARCHAR(255)`, может быть `NULL`. Внешний идентификатор назначенного мастера из корпоративной CRM. Если пусто — задача не взята в работу.
- **Индексы и связи:**
    - Индекс по `order_id` для быстрого получения всех задач конкретного заказа.

## Критерии приемки (DoD)

1. Написан и выполнен SQL-скрипт (или миграция) для создания таблицы `tasks`.
2. Внешний ключ на таблицу `orders` установлен и корректно настроено каскадное удаление.
3. Ограничения длины строк (`VARCHAR(100)`) соответствуют спецификации.
4. Таблица присутствует в БД `my_workshop`.

# Тикет 32: Настройка таблиц: создание таблицы `inventory_transactions` (Журнал складских транзакций)

## Описание

Создание таблицы-журнала для обеспечения тотального контроля за движением товарных запасов предприятия (п. 2.7.1.7 спецификации). Таблица является append-only (только добавление записей) логом всех операций: приходов от поставщиков, резервирований комплектующих и финальных списаний в производство.

## Технические требования

- **Таблица `inventory_transactions`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Системный номер записи транзакции.
    - `material_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `materials(id)` (`ON DELETE RESTRICT` — нельзя удалить материал, по которому есть история).
    - `order_id` — `INT`, _ДОПУСКАЕТ `NULL`_. Внешний ключ (FK) на таблицу `orders(id)` (`ON DELETE SET NULL`). Привязка к сделке (может быть пустым для операций "Приход", которые не привязаны к конкретному заказу).
    - `tx_type` — `VARCHAR(50)`, `NOT NULL`. Категория действия: "Приход", "Резерв", "Списание".
    - `quantity_change` — `DECIMAL(10,3)`, `NOT NULL`. Значение увеличения или уменьшения баланса (3 знака после запятой). Например: `15.500` (приход) или `-2.000` (списание).
    - `tx_date` — `DATETIME`, `NOT NULL`, По умолчанию текущее время (`DEFAULT CURRENT_TIMESTAMP`). Момент фиксации операции.
- **Индексы:**
    - Индексы по `material_id` и `order_id` для быстрой генерации складской аналитики (история движения).

## Критерии приемки (DoD)

1. Написан и выполнен SQL-скрипт (или миграция) для создания таблицы `inventory_transactions`.
2. Поле `order_id` допускает `NULL` значения.
3. Поле `quantity_change` имеет тип `DECIMAL(10,3)`.
4. Внешние ключи на `materials` и `orders` корректно функционируют.

# Тикет 33: Настройка таблиц: создание таблицы `media_files` (Медиафайлы и документы)

## Описание

Создание реестра прикрепленных цифровых материалов (п. 2.7.1.8 спецификации). Таблица не хранит сами бинарные файлы (картинки, PDF), а управляет ссылками (secure links) на сопроводительную цифровую информацию в файловом хранилище (фотографии успешного монтажа, исходные спецификации, сгенерированные чеки/акты).

## Технические требования

- **Таблица `media_files`:**
    - Движок: `InnoDB`.
- **Структура полей:**
    - `id` — `INT`, Первичный ключ (PK), Автоинкремент. Системный номер документа.
    - `order_id` — `INT`, `NOT NULL`. Внешний ключ (FK) на таблицу `orders(id)` (`ON DELETE CASCADE`). Привязка к клиентскому проекту.
    - `file_type` — `VARCHAR(50)`, `NOT NULL`. Категория загруженных данных (например, "Фото", "Акт", "Чек", "Спецификация").
    - `secure_link` — `VARCHAR(500)`, `NOT NULL`. Защищенный путь к файловому хранилищу (URL или относительный путь на сервере).
    - `uploaded_at` — `DATETIME`, `NOT NULL`, По умолчанию текущее время (`DEFAULT CURRENT_TIMESTAMP`). Дата добавления материала.
- **Индексы:**
    - Индекс по `order_id` для быстрого получения всей галереи/документов, относящихся к заказу.

## Критерии приемки (DoD)

1. Написан и выполнен SQL-скрипт (или миграция) для создания таблицы `media_files`.
2. Внешний ключ на таблицу `orders` установлен с правилом каскадного удаления (`ON DELETE CASCADE`).
3. Длина строкового поля пути к файлу (`VARCHAR(500)`) достаточна для сохранения длинных ссылок.
4. Таблица присутствует в БД `my_workshop`.
