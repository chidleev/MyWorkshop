# Тикет 61: Реализация эндпоинта расчета загруженности производственных мощностей

## Описание

Данный эндпоинт обеспечивает данными дашборд Руководителя (Фронтенд-тикет 21). Система должна агрегировать информацию из таблицы сменных заданий (`tasks`), чтобы показать распределение нагрузки по производственным линиям (распил, кромление, присадка, сборка) и выявить «узкие места» цеха.

## Технические требования

- **Маршрутизация (`src/routes/directorRoutes.ts`):**
    - Создать роут `GET /api/director/workload`.
    - Защитить через `authMiddleware` (строго только 'Руководитель').
- **Логика контроллера и SQL:**
    - Написать агрегирующий SQL-запрос для подсчета количества задач в разрезе типа операции и статуса.
    - Пример SQL: `SELECT operation_name, progress_status, COUNT(*) as task_count FROM tasks GROUP BY operation_name, progress_status`.
- **Форматирование ответа (Трансформация данных):**
    - Для удобства фронтенда преобразовать плоский массив результатов из БД в структурированный JSON-объект.
    - Пример ожидаемого ответа:
        ```json
        {
            "Распил": { "Ожидает": 15, "В работе": 2, "Завершено": 40 },
            "Кромление": { "Ожидает": 5, "В работе": 1, "Завершено": 30 }
        }
        ```

## Критерии приемки (DoD)

1. Эндпоинт `GET /api/director/workload` создан и защищен.
2. Агрегирующий SQL-запрос работает быстро и выдает корректные счетчики задач.
3. Данные трансформируются в удобный для построения графиков формат.
4. Учитываются только актуальные задачи (можно добавить фильтр по дате, например, задачи созданные за последний месяц).

# Тикет 62: Реализация «Сервиса уведомлений»: логика генерации электронных актов и чеков

## Описание

Когда заказчик получает готовую мебель (монтажник загружает фотоотчет и статус сделки меняется на «Завершен»), система должна сформировать закрывающие документы. В рамках данного тикета необходимо реализовать сервис, который «рисует» или генерирует ссылки на электронный чек и акт выполненных работ, сохраняя их в БД.

## Технические требования

- **Сервис (`src/services/NotificationService.ts` или `DocumentService.ts`):**
    - Создать сервис-подписчик, который вызывается контроллером (или `StatusManager`) при переводе `current_stage` в значение «Завершен».
- **Извлечение данных:**
    - Получить из БД `orders` и `clients` данные клиента (`email`, `full_name`), сумму (`total_cost`) и номер договора (`agreement_number`).
- **Генерация документов (Mock/Символическая):**
    - Так как генерация реальных PDF может быть избыточной на данном этапе, реализовать создание mock-ссылок:
        - `/documents/receipt_[order_id].pdf`
        - `/documents/act_[order_id].pdf`
    - Выполнить SQL-запросы: `INSERT INTO media_files (order_id, file_type, secure_link) VALUES (?, 'Чек', ?)` и то же самое для 'Акт'.

## Критерии приемки (DoD)

1. Создан модуль `NotificationService`, инкапсулирующий логику работы с документами.
2. При смене статуса заказа на «Завершен» автоматически вызывается метод генерации.
3. В таблице `media_files` успешно создаются 2 новые записи (тип "Чек" и тип "Акт") с привязкой к заказу.
4. Логика генерации обернута в `try-catch`, чтобы сбой при создании документов не откатывал успешное завершение сделки (опционально, в зависимости от строгих бизнес-правил).

# Тикет 63: Интеграция с внешним SMTP-сервером: отправка писем клиентам при статусе «Завершено»

## Описание

Завершающий шаг бизнес-процесса уведомлений (архитектура С4, взаимодействие с Почтовым сервером). После генерации акта и чека (Тикет 62), система должна отправить клиенту Email-письмо с благодарностью и ссылками на скачивание закрывающих документов.

## Технические требования

- **Зависимости:**
    - Установить пакет `nodemailer` (и `@types/nodemailer`).
- **Конфигурация (`src/config/mail.ts`):**
    - Добавить в `.env` переменные: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `MAIL_FROM`.
    - Настроить транспорт (Transporter) `nodemailer` с использованием этих переменных.
- **Логика отправки (`src/services/NotificationService.ts`):**
    - Проверить, заполнен ли `email` у клиента. Если пусто — прервать операцию (залогировать).
    - Сформировать HTML-шаблон письма:
        - _Тема:_ Ваш заказ [Номер договора] успешно завершен!
        - _Тело:_ Приветствие по имени, текст благодарности, ссылки на сгенерированные файлы (через интерполяцию строк).
    - Вызвать `transporter.sendMail()`.
- **Резервный механизм (Mock mode):**
    - Если переменные SMTP не заданы, выводить сформированное письмо в `console.log` (полезно для локальной разработки).

## Критерии приемки (DoD)

1. Пакет `nodemailer` установлен и настроен.
2. Метод отправки письма корректно интегрирован в пайплайн завершения заказа.
3. Система проверяет наличие email у клиента перед попыткой отправки.
4. Письмо успешно доставляется на тестовый ящик (или корректно логируется в консоль при локальной разработке).
5. Сетевые ошибки SMTP-сервера перехватываются (`catch`) и не вызывают крах приложения Node.js.
