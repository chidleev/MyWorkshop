# Тикет 61: Реализация эндпоинта расчета загруженности производственных мощностей

## Описание

Данный эндпоинт обеспечивает данными дашборд Руководителя (Фронтенд-тикет 21). Система должна агрегировать информацию из таблицы сменных заданий (`tasks`), чтобы показать распределение нагрузки по производственным линиям (распил, кромление, присадка, сборка) и выявить «узкие места» цеха.

## Технические требования

- **Маршрутизация (`src/routes/directorRoutes.ts`):**
    - Создать роут `GET /api/director/workload`.
    - Защитить через `authMiddleware` (строго только 'Руководитель').
- **Логика контроллера и SQL:**
    - Написать агрегирующий SQL-запрос для подсчета количества задач в разрезе типа операции и статуса.
    - Пример SQL: `SELECT operation_name, progress_status, COUNT(*) as task_count FROM tasks GROUP BY operation_name, progress_status`.
- **Форматирование ответа (Трансформация данных):**
    - Для удобства фронтенда преобразовать плоский массив результатов из БД в структурированный JSON-объект.
    - Пример ожидаемого ответа:
        ```json
        {
            "Распил": { "Ожидает": 15, "В работе": 2, "Завершен": 40 },
            "Кромление": { "Ожидает": 5, "В работе": 1, "Завершен": 30 }
        }
        ```

## Критерии приемки (DoD)

1. Эндпоинт `GET /api/director/workload` создан и защищен.
2. Агрегирующий SQL-запрос работает быстро и выдает корректные счетчики задач.
3. Данные трансформируются в удобный для построения графиков формат.
4. Учитываются все строки `tasks` в текущей реализации (отдельный фильтр по дате на бэкенде не задан).

# Тикет 62: Реализация «Сервиса уведомлений»: логика генерации электронных актов и чеков

## Описание

Когда заказчик получает готовую мебель (монтажник загружает фотоотчет и статус сделки меняется на «Завершен»), система должна сформировать закрывающие документы. Реализована полноценная генерация PDF и запись ссылок в БД.

## Технические требования

- **Точка входа (`src/services/NotificationService.ts`, метод `generateDocuments`):**
    - Вызывается из `StatusManager.changeStatus` **после** успешного `commit` транзакции при переходе в «Завершен» (ошибки пайплайна документов/почты логируются и не откатывают статус заказа).
- **Сервис генерации PDF (`src/services/ClosingDocumentsService.ts`):**
    - Загрузка контекста: заказ + клиент (`orders` + `clients`), строки спецификации (`specification_items` + `materials`) для таблицы в чеке.
    - Генерация двух файлов **pdfmake** (шрифты Roboto из `node_modules/pdfmake/fonts/Roboto`, кириллица), запись на диск в `documentsPath`: `receipt_{orderId}.pdf`, `act_{orderId}.pdf`.
    - Содержимое: **чек** — реквизиты заказчика, таблица позиций, итог по `total_cost`; **акт** — текст сдачи-приёмки по изготовлению мебели, сумма, блоки под подписи.
- **Публичные URL (совпадают с `secure_link` в БД):**
    - `/documents/receipt_[order_id].pdf`
    - `/documents/act_[order_id].pdf`
- **`media_files`:**
    - Перед вставкой: `DELETE ... WHERE order_id = ? AND secure_link IN (?, ?)` для идемпотентности при повторной генерации.
    - Затем `INSERT` двух строк (типы «Чек» и «Акт»).
- **Раздача:** `app.use('/documents', express.static(documentsPath))` — файлы реально отдаются по указанным URL.
- **Тесты:** `backend/tests/closingDocumentsPdf.test.ts` — генерация во временную папку, проверка `%PDF`, запрос `GET /documents/receipt_....pdf` через `supertest`.

## Критерии приемки (DoD)

1. Создан модуль `NotificationService`, инкапсулирующий оркестрацию документов и почты; генерация вынесена в `ClosingDocumentsService`.
2. При смене статуса заказа на «Завершен» автоматически вызывается метод генерации (после коммита статуса).
3. В каталоге `documents/` появляются два PDF-файла; в `media_files` — две актуальные записи с привязкой к заказу.
4. Сбой генерации/почты не откатывает перевод в «Завершен» (`try/catch` в `StatusManager` вокруг пайплайна уведомлений).
5. Закрывающие документы доступны по HTTP (`Content-Type` с `application/pdf`).

# Тикет 63: Интеграция с внешним SMTP-сервером: отправка писем клиентам при статусе «Завершено»

## Описание

Завершающий шаг бизнес-процесса уведомлений (архитектура С4, взаимодействие с Почтовым сервером). После генерации акта и чека (Тикет 62), система отправляет клиенту письмо с благодарностью и ссылками на скачивание закрывающих документов.

## Технические требования

- **Зависимости:**
    - Пакет `nodemailer` (и `@types/nodemailer`).
- **Конфигурация (`src/config/mail.ts`):**
    - Переменные окружения: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `MAIL_FROM`.
    - Транспорт `nodemailer` при наличии всех полей SMTP; иначе `null` (mock).
- **Логика отправки (`src/services/NotificationService.ts`):**
    - Проверка `email` у клиента; при пустом — лог и выход.
    - Письмо о завершении заказа: HTML + plain text из **`src/utils/transactionalEmailHtml.ts`** (`buildOrderCompletedEmail`): табличная вёрстка, прехедер, кнопки «Скачать чек/акт», дублирование ссылок текстом, экранирование пользовательских полей.
    - Абсолютные ссылки в письме: базовый origin из **`FRONTEND_URL`** или первого значения **`FRONTEND_URLS`** (см. `absoluteUrl` в том же модуле); без настройки — относительные пути (как в БД).
    - _Тема (факт):_ «Ваш заказ [номер/договор] успешно завершён».
    - `transporter.sendMail({ html, text, ... })`.
- **Письмо об отклонении веб-заявки:** `sendWebApplicationRejectedEmail` — тот же модуль шаблонов (`buildWebApplicationRejectedEmail`), HTML + text.
- **Резервный механизм (Mock mode):**
    - При отсутствии SMTP — вывод содержимого письма в `console.log` (включая `text`).

## Критерии приемки (DoD)

1. Пакет `nodemailer` установлен и настроен.
2. Метод отправки письма корректно интегрирован в пайплайн завершения заказа (после генерации PDF).
3. Система проверяет наличие email у клиента перед попыткой отправки.
4. Письмо доставляется на тестовый ящик или логируется в консоль при локальной разработке; в письме осмысленный HTML и текстовая версия.
5. Сетевые ошибки SMTP перехватываются (`catch`) и не роняют процесс Node.js.
