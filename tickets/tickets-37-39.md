# Тикет 37: Реализация модуля «Адаптер CRM»: HTTP-клиент для запросов к внешней CRM

## Описание

Согласно архитектуре системы (п. 2.1.3.1 спецификации), профили сотрудников и их роли хранятся во внешней мастер-системе (CRM). Серверному приложению требуется HTTP-шлюз (Адаптер) для связи с этой системой. Адаптер будет передавать токен, полученный от клиента, на сервер CRM и получать в ответ информацию о валидности сессии, роли и ID пользователя.

## Технические требования

- **Служба (`src/services/CrmAdapter.ts`):**
    - Создать класс или набор утилит для выполнения HTTP-запросов (рекомендуется использовать встроенный `fetch` или легковесный `axios`).
    - Реализовать метод `verifyToken(token: string): Promise<UserProfile>`.
- **Конфигурация и Mock-режим:**
    - Добавить переменную окружения `CRM_API_URL`.
    - _Важно:_ Так как реального доступа к CRM на этапе разработки может не быть, необходимо реализовать mock-режим. Если `CRM_API_URL` не задан (или установлен флаг `USE_MOCK_CRM=true`), Адаптер должен локально разбирать токен фронтенда: префикс **`mock:`**, затем ключ роли (`manager`, `master`, `installer`, `storekeeper`, `buyer`, `director`), затем **`employee_ext_id`** до конца строки (например `mock:manager:crm-manager-001`). Для обратной совместимости допускается разбор старых токенов без префикса `mock:` (fallback в `buildMockProfile`).
- **Интерфейсы (TypeScript):**
    - Описать интерфейс `UserProfile`, содержащий поля `id` (соответствует `manager_ext_id` / `employee_ext_id` в БД), `fullName`, `role`.

## Критерии приемки (DoD)

1. Создан сервис `CrmAdapter.ts` с методом `verifyToken`.
2. Реализован mock-режим для локального тестирования ролевой модели.
3. Метод корректно обрабатывает сетевые ошибки (если внешний сервер недоступен) и бросает исключения (например, `AppError(503)` или `AppError(401)`).

# Тикет 38: Реализация Middleware: Верификация токена и проверка прав (ролей) сотрудника

## Описание

Любые закрытые API-эндпоинты должны быть защищены от несанкционированного доступа. Необходимо создать промежуточное ПО (middleware), которое будет перехватывать входящие HTTP-запросы, извлекать токен авторизации и проверять права пользователя через Адаптер CRM.

## Технические требования

- **Auth Middleware (`src/middlewares/authMiddleware.ts`):**
    - Извлечь токен из заголовка `Authorization: Bearer <token>`.
    - Если токен отсутствует — выбросить `AppError(401, 'Токен не предоставлен')`.
    - Передать токен в `CrmAdapter.verifyToken()`. При невалидном токене — `AppError(401, 'Недействительный токен')`.
    - В случае успеха записать данные профиля в объект запроса (`req.user = profile`) для дальнейшего использования в контроллерах.
- **Role Middleware (`src/middlewares/roleMiddleware.ts`):**
    - Реализовать функцию-фабрику `requireRole(allowedRoles: string[])`.
    - Middleware должен проверять, совпадает ли `req.user.role` с разрешенным списком.
    - Если совпадения нет — выбрасывать `AppError(403, 'Нет доступа')`.
- **Расширение типов (TypeScript):**
    - Расширить интерфейс `Request` из пространства имен Express, чтобы TypeScript распознавал свойство `req.user`.

## Критерии приемки (DoD)

1. Реализован `authMiddleware`, корректно извлекающий Bearer токен.
2. Реализован `roleMiddleware` для ограничения доступа по ролям.
3. Ошибки авторизации (401) и отсутствия прав (403) обрабатываются глобальным обработчиком (из Тикета 36) и возвращаются в виде стандартизированного JSON.
4. Настроен TypeScript definition (`custom.d.ts` или расширение интерфейса `Request`) для поля `req.user`.

# Тикет 39: Реализация «Контроллера заказов»: POST /api/orders (Создание заказа)

## Описание

Первая бизнес-операция в системе — заведение новой карточки заказа в базе данных (п. 2.1.3.2). Эндпоинт будет использоваться менеджерами через веб-интерфейс, а также может выступать точкой приема лидов с внешнего сайта.

## Технические требования

- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - Создать роут `POST /api/orders`.
    - Защитить `authMiddleware` и `requireRole(['Менеджер', 'Руководитель'])`.
- **Контроллер (`src/controllers/OrderController.ts`):**
    - Тело (Zod `createOrderSchema`): `full_name`, `phone` (строка, может быть пустой), **`email` (обязательный валидный email)**, опционально `address`, `agreement_number`, `target_date` (строка для `DATETIME`), опционально `manager_ext_id` — для роли **`Руководитель`** позволяет указать внешний id менеджера-владельца; для **`Менеджер`** владелец всегда `req.user.id`.
    - SQL-транзакция: `INSERT INTO clients (...)`, затем `INSERT INTO orders (..., total_cost, current_stage) VALUES (..., 0.00, 'Новый')`, при пустом номере договора — `UPDATE` с `formatAutoAgreementNumber`, запись в `status_histories` через `StatusManager.appendHistory`.
- **Ответ:** HTTP 201, `{ status: 'success', data: { id, client_id } }`.

## Критерии приемки (DoD)

1. Создан эндпоинт `POST /api/orders`.
2. Роут защищен проверкой токена и ролевой моделью.
3. Присутствует валидация входящего JSON, возвращающая HTTP 400 при отсутствии обязательных параметров.
4. При успешном выполнении в БД создается новая запись, а сервер возвращает HTTP 201 (Created) с ID созданного заказа.
