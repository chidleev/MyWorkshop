# Тикет 40: Реализация «Контроллера заказов»: GET /api/orders (Получение списка заказов с фильтрами)

## Описание

Для работы канбан-доски менеджера и глобального дашборда руководителя сервер должен предоставлять список заказов. Эндпоинт должен поддерживать гибкую фильтрацию и учитывать ролевую модель: менеджер по умолчанию видит только свои сделки, а руководитель — сделки всей мастерской.

## Технические требования

- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - Создать роут `GET /api/orders`.
    - Защитить эндпоинт с помощью `authMiddleware`.
- **Логика контроллера (`src/controllers/OrderController.ts`):**
    - Извлечь параметры запроса (query params): `status`, `manager_id`, `target_date`.
    - **Ролевой доступ (Scoping):**
        - Если `req.user.role === 'Менеджер'`, принудительно добавить в SQL-запрос условие `WHERE orders.manager_ext_id = ?` (использовать `req.user.id`).
        - Если `req.user.role === 'Руководитель'`, применять фильтр по `manager_id` только если он явно передан в query-параметрах.
- **SQL-запрос:**
    - Реализовать запрос с использованием `JOIN` таблицы `clients`, чтобы в ответе сразу возвращались имя клиента (`full_name`) и телефон (`phone`), необходимые для отрисовки карточек на клиенте.
- **Сортировка:**
    - По умолчанию сортировать по `target_date` (по возрастанию) или `created_at` (по убыванию).

## Критерии приемки (DoD)

1. Создан эндпоинт `GET /api/orders`.
2. Ролевая изоляция работает: менеджер не может получить список заказов другого менеджера.
3. Фильтры (по статусу и датам) корректно транслируются в SQL-условия (`WHERE`).
4. В JSON-ответе присутствуют данные как самого заказа, так и связанного клиента (через `JOIN`).

# Тикет 41: Реализация «Контроллера заказов»: PUT/DELETE /api/orders/:id (Редактирование/удаление)

## Описание

Функционал управления жизненным циклом сделки на ранних этапах. Менеджер может скорректировать плановые сроки, изменить номер договора или полностью отменить (удалить) карточку заказа, если клиент отказался до начала работ.

## Технические требования

- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - Создать роуты `PUT /api/orders/:id` и `DELETE /api/orders/:id`.
    - Защитить через `authMiddleware` и `requireRole(['Менеджер', 'Руководитель'])`.
- **Права доступа к конкретной записи:**
    - Перед выполнением UPDATE или DELETE контроллер должен сделать `SELECT manager_ext_id FROM orders WHERE id = ?`.
    - Если заказ принадлежит другому сотруднику (а текущий юзер — не Руководитель), выбрасывать `AppError(403, 'Вы не можете изменить чужой заказ')`.
- **Логика PUT:**
    - Валидация входящего body. Допустимо обновлять: `agreement_number`, `target_date`.
    - Опционально: если переданы данные клиента (`full_name`, `phone`), обновлять связанную запись в таблице `clients` (использовать SQL-транзакцию для консистентности).
- **Логика DELETE:**
    - Выполнить `DELETE FROM orders WHERE id = ?`. Каскадное удаление спецификаций и задач отработает на уровне БД (настроено в Тикетах 29, 31).

## Критерии приемки (DoD)

1. Эндпоинты обновления и удаления созданы и работают.
2. Реализована строгая проверка прав владения (ownership): менеджер может удалить только свой заказ.
3. При запросе с несуществующим ID возвращается `AppError(404, 'Заказ не найден')`.
4. В случае PUT-запроса изменения успешно фиксируются в базе данных.

# Тикет 42: Реализация эндпоинта для приема заявок с внешнего сайта через API

## Описание

Обеспечение интеграции с внешними системами (корпоративный сайт, лендинги). Когда клиент заполняет форму обратной связи на сайте (Фронтенд-тикет 23), запрос уходит на публичный или полупубличный эндпоинт API, который создает карточку "Лида" (Заказа в статусе "Новый").

## Технические требования

- **Маршрутизация (`src/routes/externalRoutes.ts`):**
    - Создать отдельный роут, например `POST /api/external/leads`.
- **Безопасность (API-ключ):**
    - Поскольку запрос приходит не от авторизованного сотрудника (нет CRM-токена), эндпоинт не должен использовать `authMiddleware`.
    - Написать новый `apiKeyMiddleware`, который проверяет наличие кастомного заголовка (например, `X-API-KEY`) и сверяет его с переменной окружения `EXTERNAL_API_KEY`. При несовпадении — 401 Unauthorized.
- **Логика контроллера:**
    - Принимает JSON: `client_name`, `client_phone`, `comment`.
    - Начать SQL-транзакцию:
        1.  `INSERT INTO clients (full_name, phone) VALUES (...)` -> получить `client_id`.
        2.  `INSERT INTO orders (client_id, manager_ext_id, current_stage, total_cost) VALUES (...)`.
    - _Специфика БД:_ Поле `manager_ext_id` является `NOT NULL`. Для веб-заявок использовать системный константный ID (например, `'system_web_lead'`), чтобы такие заказы попадали в статус нераспределенных. Сгенерировать временный `agreement_number` (например, `WEB-{TIMESTAMP}`).
    - Закоммитить транзакцию.

## Критерии приемки (DoD)

1. Создан выделенный эндпоинт для внешних систем, защищенный статичным API-ключом (не CRM-токеном).
2. Запрос успешно создает новую запись в таблице `clients` и привязанную к ней запись в `orders`.
3. Поле `manager_ext_id` заполняется специальным системным идентификатором.
4. Использование транзакции (`BEGIN`, `COMMIT`, `ROLLBACK`) гарантирует, что клиент не будет создан, если вставка заказа завершится ошибкой.
