# Тикет 40: Реализация «Контроллера заказов»: список заказов, создание заказа, веб-заявки, вспомогательные поля для UI

## Описание

Для канбан-доски менеджера, экрана веб-заявок и интеграций сервер предоставляет REST API заказов. Менеджер по умолчанию видит только свои сделки; руководитель при использовании общего `GET /api/orders` не ограничивается менеджером, если не передан `manager_id`. Отдельно выделены веб-заявки с сайта (`manager_ext_id = 'system_web_lead'`, стадия `'Новый'`).

## Технические требования

- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - `POST /api/orders` — создание заказа: `authMiddleware`, роли `Менеджер`, `Руководитель`. Тело (zod `createOrderSchema`): обязательные `full_name`, валидный `email`; `phone` — строка (может быть пустой); опционально `address`, `agreement_number`, `target_date` (ISO-строка для `DATETIME`), `manager_ext_id` (только смысл для роли `Руководитель` — под кого создать заказ; иначе берётся `req.user.id`). Заказ создаётся в статусе `'Новый'`, `total_cost = 0.00`; при пустом `agreement_number` генерируется `MM-YYYY-NNNN`; в `status_histories` пишется первая запись стадии.
    - `GET /api/orders` — список: `authMiddleware` без жёсткого `requireRole` (доступ по политике токена; на практике используют менеджер и руководитель). Query: `status`, `manager_id` (только для руководителя), `target_date` (точное совпадение с полем `orders.target_date`). Для менеджера всегда добавляется `o.manager_ext_id = req.user.id`.
    - `GET /api/orders/web-applications` — только `Менеджер`: заказы с `manager_ext_id = 'system_web_lead'` и `current_stage = 'Новый'`, с полями клиента и агрегатами по спецификации/задачам (как в списке заказов, с `production_stage = NULL`).
    - `POST /api/orders/web-applications/:id/claim` — только `Менеджер`: закрепление веб-заявки за текущим менеджером, обновление клиента и заказа (в т.ч. `agreement_number`, `target_date`), транзакция БД.
    - `POST /api/orders/web-applications/:id/reject` — только `Менеджер`: отклонение заявки; после успешного отклонения вызывается `NotificationService.sendWebApplicationRejectedEmail` (HTML + text из `transactionalEmailHtml.ts`; SMTP или mock-лог в консоль — см. Тикет 63).
- **Расширенный SELECT для `GET /api/orders` (и общая форма DTO):**
    - `JOIN clients`: `full_name`, `phone`, `email`, `address`.
    - Вычисляемое поле `production_stage` — имя операции из `tasks`: сначала первая задача в статусе `'В работе'`, иначе первая в `'Ожидает'`, иначе `NULL`.
    - Флаги/счётчики: `has_specification`, `tasks_total_count`, `tasks_started_count`, `tasks_completed_count` (подзапросы по `specification_items` и `tasks`).
- **Сортировка:** `ORDER BY o.target_date ASC, o.created_at DESC`.
- **Примечание:** фильтр `is_overdue` для менеджерского списка в текущей реализации на бэкенде **не** обрабатывается; просрочку для менеджера при необходимости считает клиент. Для руководителя глобальный список и просрочка — см. Тикет 58 (`GET /api/director/orders`).

## Критерии приемки (DoD)

1. Работают `POST /api/orders` и `GET /api/orders` с ролевой изоляцией менеджера.
2. Реализованы `GET/POST` маршруты веб-заявок с проверкой роли `Менеджер` и условий `system_web_lead` / `'Новый'` где применимо.
3. В ответе списка заказов есть поля клиента, производственного этапа и счётчиков задач/спецификации.
4. Фильтры `status`, `manager_id` (для руководителя), `target_date` корректно формируют `WHERE`.

# Тикет 41: Реализация «Контроллера заказов»: PUT/DELETE /api/orders/:id (Редактирование/удаление)

## Описание

Функционал управления жизненным циклом сделки на ранних этапах. Менеджер может скорректировать плановые сроки, изменить номер договора или полностью отменить (удалить) карточку заказа, если клиент отказался до начала работ.

## Технические требования

- **Маршрутизация (`src/routes/orderRoutes.ts`):**
    - Создать роуты `PUT /api/orders/:id` и `DELETE /api/orders/:id`.
    - Защитить через `authMiddleware` и `requireRole(['Менеджер', 'Руководитель'])`.
- **Права доступа к конкретной записи:**
    - Перед выполнением UPDATE или DELETE контроллер должен сделать `SELECT manager_ext_id FROM orders WHERE id = ?`.
    - Если заказ принадлежит другому сотруднику (а текущий юзер — не Руководитель), выбрасывать `AppError(403, 'Вы не можете изменить чужой заказ')`.
- **Логика PUT:**
    - Валидация входящего body. Допустимо обновлять: `agreement_number`, `target_date` (строка для `DATETIME`), контактные поля клиента (`full_name`, `phone`, `email`, `address`) по фактической реализации контроллера.
    - Обновление строки заказа должно сопровождаться обновлением `updated_at` (если не сделано триггером СУБД — на уровне приложения).
- **Отдельный маршрут наценки:** `PATCH /api/orders/:id/pricing-markup` (`Менеджер`, `Руководитель`) — тело: `pricing_markup_mode`, `pricing_markup_value`; пересчитывается `total_cost` от суммы `sale_price` по строкам спецификации (материальный субтотал) и текущего режима наценки; разрешено **только** для заказа в статусе `'Новый'` (иначе 409).
- **Логика DELETE:**
    - Выполнить `DELETE FROM orders WHERE id = ?`. Каскадное удаление спецификаций и задач отработает на уровне БД (настроено в Тикетах 29, 31).

## Критерии приемки (DoD)

1. Эндпоинты обновления и удаления созданы и работают.
2. Реализована строгая проверка прав владения (ownership): менеджер может удалить только свой заказ.
3. При запросе с несуществующим ID возвращается `AppError(404, 'Заказ не найден')`.
4. В случае PUT-запроса изменения успешно фиксируются в базе данных.

# Тикет 42: Реализация эндпоинта для приема заявок с внешнего сайта через API

## Описание

Обеспечение интеграции с внешними системами (корпоративный сайт, лендинги). Когда клиент заполняет форму обратной связи на сайте (Фронтенд-тикет 23), запрос уходит на публичный эндпоинт API, который создает карточку "Лида" (Заказа в статусе "Новый").

## Технические требования

- **Маршрутизация (`src/routes/externalRoutes.ts`):**
    - Роут `POST /api/external/leads` без `authMiddleware` (лид создаётся от анонимного сайта).
- **Безопасность (опционально для усиления):**
    - В репозитории есть заготовка `src/middlewares/apiKeyMiddleware.ts` (заголовок `X-API-KEY` ↔ `EXTERNAL_API_KEY`), но **в текущей сборке** к `POST /api/external/leads` middleware **не подключён** — доступ ограничивается CORS и валидацией тела. При необходимости жёсткой защиты — подключить middleware к роутеру `externalRoutes`.
- **Логика контроллера (`ExternalController.createLead`):**
    - Zod-схема после `preprocess`: обязательные `client_name`, `client_email` (email); опциональные `client_phone`, `comment`; допускается дублирующее поле `email` в теле — мапится в `client_email`.
    - SQL-транзакция:
        1.  `INSERT INTO clients (full_name, phone, email, address) VALUES (...)` — `phone` из заявки или `NULL`; **`address`** при наличии непустого `comment` заполняется строкой вида `Комментарий заявки: <текст>`, иначе `NULL`.
        2.  `INSERT INTO orders (...)` с `manager_ext_id = 'system_web_lead'`, `agreement_number = WEB-{timestamp}`, `total_cost = 0.00`, `current_stage = 'Новый'`.
    - Запись в `status_histories` через `StatusManager.appendHistory` с `employee_ext_id = 'system_web_lead'`.
    - Ответ HTTP 201: `{ status: 'success', data: { client_id, order_id, agreement_number } }`.

## Критерии приемки (DoD)

1. Создан эндпоинт `POST /api/external/leads`, создающий клиента и заказ в одной транзакции.
2. Валидация тела запроса; при ошибках — HTTP 400.
3. Поле `manager_ext_id` заказа — `'system_web_lead'`.
4. Использование транзакции (`BEGIN`, `COMMIT`, `ROLLBACK`) гарантирует целостность данных.
