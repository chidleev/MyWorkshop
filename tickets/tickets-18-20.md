# Тикет 18: Разработка экрана истории транзакций и зарезервированных материалов

## Описание

Поскольку система автоматически резервирует и списывает материалы при переходе заказа в стадию производства, Кладовщику необходим инструмент для наблюдения за этим потоком. На данном экране он сможет отслеживать, под какой именно шкаф или кухню была заблокирована конкретная фурнитура или списан плитный материал.

## Технические требования

- **Страница (`src/views/Storekeeper/TransactionsView.vue`):**
    - Разместить ссылку на страницу в сайдбаре для роли «Кладовщик».
    - Журнал на **`AppDataTable`**: строки из ответа `GET /api/inventory/transactions` (поля с джойном к номенклатуре на бэкенде).
- **Колонки (суть данных):** дата/время (`tx_date`), артикул и имя материала, тип (`tx_type`: Приход / Резерв / Списание), изменение (`quantity_change`), `order_id` (может быть пустым для прихода).
- **Стейт / API:**
    - `fetchTransactions({ tx_type, order_id, limit, offset })` → `GET /api/inventory/transactions`.
- **Фильтрация:**
    - Добавить фильтр по типу действия (только приходы / только резервы).
    - Поиск по номеру заказа (`order_id`).
- **SPA / UX:** при смене фильтров или строки поиска после первой загрузки экрана обновлять журнал **мягко**: таблица не исчезает под полноэкранным «Загрузка журнала…» — использовать фоновый индикатор (например `loadTransactions({ soft: true })`, `isRefreshing`, текст «Обновление журнала…»). Полный блок загрузки — только при первом открытии страницы.

## Критерии приемки (DoD)

1. Таблица журнала строится из ответа API.
2. Цветовая индикация изменений: приход зеленого цвета (+), списание/резерв красного (-).
3. Работает фильтрация по номеру заказа и типу операции.
4. Экран доступен только для авторизованных кладовщиков.
5. Смена фильтров не создаёт ощущения «перезагрузки» всей страницы (мягкое обновление списка).

# Тикет 19: Разработка экрана аналитики дефицитных позиций (Закупщик)

## Описание

Рабочее пространство для роли «Закупщик». Из-за автоматического резервирования комплектующих под новые сделки, виртуальный баланс некоторых складских позиций может уйти в минус. Закупщик регулярно обращается к цифровой панели для получения списка отсутствующей фурнитуры, чтобы сформировать заказы поставщикам.

## Технические требования

- **Страница (`src/views/Buyer/DeficitReportView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Закупщик'`.
    - Перед запросом к API — проверка доступности бэкенда через `isServerAvailable` / `SERVER_UNAVAILABLE_MESSAGE` из `src/utils/serverHealth.ts` (без лишних запросов при «сервер недоступен»).
    - Таблица на базе **`AppDataTable`** (`src/components/Common/AppDataTable.vue`) с колонками артикул, наименование, «К закупке» (редактируемое количество поверх `suggested_qty` с дефолтом из ответа API), цена закупки (`base_cost`), сумма строки (`purchase_qty * base_cost`).
- **Логика отображения данных:**
    - `fetchDeficitReport({ include_reserves })` → `GET /api/inventory/deficit` с query `include_reserves` (см. Тикет 56); переключатель на UI при изменении перезагружает отчёт.
    - Итог **`total_budget`**: сумма по отредактированным строкам на клиенте (`computed` по `displayRows`); при желании сверять с корнем ответа API как с начальным ориентиром.
- **SPA / UX:** при переключении «Учитывать резервы», по кнопке «Обновить» и при повторных запросах, когда таблица уже отображалась, не скрывать весь блок отчёта под полноэкранным «Загрузка отчета…» — использовать мягкое обновление (например `loadDeficitReport({ soft: true })`, индикатор «Обновление отчёта…»). Полный скелетон — при первом заходе на экран или при отсутствии успешно загруженных данных.
- **Экспорт:**
    - Кнопка «Сформировать заявку / Экспорт в CSV» — клиентская генерация файла `deficit-report.csv` (разделитель `;`, BOM `\uFEFF`), колонки: артикул, наименование, к закупке, цена закупки, сумма; при пустом списке — `showWarning` из `notification.ts`.

## Критерии приемки (DoD)

1. Создан защищенный роут `/buyer/deficit`. При логине под Закупщиком происходит редирект на этот экран.
2. В таблице только позиции, возвращённые сервером как дефицитные (по правилу `include_reserves` / физического остатка).
3. Система выводит бюджет (сумма по строкам с учётом правок количеств на клиенте) и корректно обрабатывает `include_reserves`.
4. Экспорт в CSV работает в браузере; интерфейс остаётся читаемым аналитическим отчётом.
5. Повторная загрузка отчёта (чекбокс резервов, «Обновить») не имитирует полную перезагрузку SPA: таблица остаётся на экране с лёгким индикатором обновления.

# Тикет 20: Разработка главного табло глобального мониторинга всех активных сделок (Руководитель)

## Описание

Для предотвращения срывов сроков Руководитель должен видеть «картину сверху». Ему необходим сводный реестр всех текущих проектов мастерской. Одно нажатие кнопки должно выводить на экран полный список сделок вместе с их актуальными статусами готовности, независимо от того, какой менеджер их ведет.

## Технические требования

- **Страница (`src/views/Director/GlobalMonitoringView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Предпочтительный формат вывода: **Масштабная Канбан-доска** или плотная Data Table с расширенными фильтрами.
- **Отличие от экрана Менеджера (Тикет 6):**
    - В отличие от Менеджера, который видит только _свои_ заказы, Руководитель видит заказы _всех_ менеджеров.
    - В карточке заказа обязательно должно выводиться ФИО/ID ответственного менеджера (`manager_ext_id`).
- **Представление данных:** на широких экранах — таблица **`AppDataTable`** с фильтрами; на узких — сохранить читаемый мобильный вид (карточки), если реализовано в компоненте.
- **Индикаторы (Warnings):**
    - Добавить логику визуального выделения просроченных сделок (где `target_date` меньше текущей даты, но статус не «Завершен»). Такие карточки должны подсвечиваться красным контуром или мигающим бейджем.
- **Фильтры:**
    - Добавить фильтр по ответственному менеджеру и по статусу проекта.
- **Данные:** `fetchGlobalOrders(params)` → `GET /api/director/orders` с query `manager_id`, `status`, `is_overdue`.
- **Детализация:** переход на роут `/director/orders/:id` (`DirectorOrderDetailView`) с загрузкой `fetchDirectorOrderDetails` (`GET /api/director/orders/:id`).

## Критерии приемки (DoD)

1. Создан роут `/director/monitoring` (только для Руководителя).
2. На экране отображаются данные `GET /api/director/orders`, без mock-массива.
3. Карточки заказов отображают ответственного менеджера.
4. Корректно работает подсветка просроченных заказов (исходя из даты `target_date`).
5. Работают глобальные фильтры по статусам и менеджерам (включая запрос с `is_overdue=true`).
