# Тикет 18: Разработка экрана истории транзакций и зарезервированных материалов

## Описание

Поскольку система автоматически резервирует и списывает материалы при переходе заказа в стадию производства, Кладовщику необходим инструмент для наблюдения за этим потоком. На данном экране он сможет отслеживать, под какой именно шкаф или кухню была заблокирована конкретная фурнитура или списан плитный материал.

## Технические требования

- **Страница (`src/views/Storekeeper/TransactionsView.vue`):**
    - Разместить ссылку на страницу в сайдбаре для роли «Кладовщик».
    - Отображение данных: таблица, имитирующая структуру `inventory_transactions` из БД.
- **Колонки таблицы:**
    - Дата и время операции (`tx_date`).
    - Номенклатура (связь с `materials` — Артикул и Наименование).
    - Тип действия (`tx_type`): Приход, Резерв, Списание.
    - Изменение баланса (`quantity_change`).
    - Сделка / Заказ (`order_id`) — чтобы понимать, куда ушел материал (для прихода может быть пустым).
- **Стейт / Mock-данные:**
    - Создать mock-массив истории транзакций.
- **Фильтрация:**
    - Добавить фильтр по типу действия (только приходы / только резервы).
    - Поиск по номеру заказа (`order_id`).

## Критерии приемки (DoD)

1. Создана таблица журнала транзакций, корректно выводящая мок-данные.
2. Цветовая индикация изменений: приход зеленого цвета (+), списание/резерв красного (-).
3. Работает фильтрация по номеру заказа и типу операции.
4. Экран доступен только для авторизованных кладовщиков.

# Тикет 19: Разработка экрана аналитики дефицитных позиций (Закупщик)

## Описание

Рабочее пространство для роли «Закупщик». Из-за автоматического резервирования комплектующих под новые сделки, виртуальный баланс некоторых складских позиций может уйти в минус. Закупщик регулярно обращается к цифровой панели для получения списка отсутствующей фурнитуры, чтобы сформировать заказы поставщикам.

## Технические требования

- **Страница (`src/views/Buyer/DeficitReportView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Закупщик'`.
    - Сверстать таблицу или список «Отчет о дефиците материалов».
- **Логика отображения данных:**
    - Использовать общий mock-стейт `materials` (складская номенклатура).
    - Отфильтровать массив: выводить **только** те позиции, где физический остаток (`current_stock`) меньше нуля ( `< 0` ).
- **Колонки отчета:**
    - Артикул и Наименование детали.
    - Текущий отрицательный баланс (показывать по модулю как «Требуется закупить», например: если баланс -5, писать "К закупке: 5").
    - Базовая стоимость закупки (`base_cost` из справочника) и итоговая сумма закупки (Требуется \* Базовая стоимость).
- **UI элементы:**
    - Добавить кнопку-заглушку «Экспорт в Excel / PDF» или «Сформировать заявку».

## Критерии приемки (DoD)

1. Создан защищенный роут `/buyer/deficit`. При логине под Закупщиком происходит редирект на этот экран.
2. Таблица строго фильтрует данные и показывает только дефицитные позиции (с отрицательным остатком).
3. Система автоматически вычисляет и выводит бюджет, необходимый для покрытия дефицита.
4. Интерфейс выглядит как строгий аналитический отчет.

# Тикет 20: Разработка главного табло глобального мониторинга всех активных сделок (Руководитель)

## Описание

Для предотвращения срывов сроков Руководитель должен видеть «картину сверху». Ему необходим сводный реестр всех текущих проектов мастерской. Одно нажатие кнопки должно выводить на экран полный список сделок вместе с их актуальными статусами готовности, независимо от того, какой менеджер их ведет.

## Технические требования

- **Страница (`src/views/Director/GlobalMonitoringView.vue`):**
    - Доступ к роуту разрешен только для `userRole === 'Руководитель'`.
    - Предпочтительный формат вывода: **Масштабная Канбан-доска** или плотная Data Table с расширенными фильтрами.
- **Отличие от экрана Менеджера (Тикет 6):**
    - В отличие от Менеджера, который видит только _свои_ заказы, Руководитель видит заказы _всех_ менеджеров.
    - В карточке заказа обязательно должно выводиться ФИО/ID ответственного менеджера (`manager_ext_id`).
- **Индикаторы (Warnings):**
    - Добавить логику визуального выделения просроченных сделок (где `target_date` меньше текущей даты, но статус не «Завершен»). Такие карточки должны подсвечиваться красным контуром или мигающим бейджем.
- **Фильтры:**
    - Добавить фильтр по ответственному менеджеру и по статусу проекта.

## Критерии приемки (DoD)

1. Создан роут `/director/monitoring` (Только для Руководителя).
2. На экране корректно рендерится полный массив mock-сделок всех сотрудников.
3. Карточки заказов отображают ответственного менеджера.
4. Корректно работает подсветка просроченных заказов (исходя из даты `target_date`).
5. Работают глобальные фильтры по статусам и менеджерам.
